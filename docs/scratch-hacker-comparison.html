Real-world Java patch efficiencySpring/Java enterprise manifest only

Same dataset (175 events), two clean time scopes — does the discriminator survive 7-year scaling?

Scope: real-world Java enterprise manifest, all 11 actually-exploited events. This page uses the data/seven-year-manifest-events.json dataset — a curated representative enterprise Java estate (Spring Framework, Spring Security, Spring Boot, Tomcat, Jackson, XStream, log4j, ActiveMQ broker, snakeyaml, commons-text, paramiko-style auxiliaries, ~40 transitive deps). 175 critical/high CVEs over 7 years; 11 actually exploited per KEV ∪ Metasploit ∪ ExploitDB.

We have abandoned the published "8 in-scope" filter that excluded 3 exploited events as out-of-NP+DI-design-scope (commons-text, SnakeYAML, ActiveMQ OpenWire). Those are real exploits that defenders care about; the in-scope filter was methodologically defensible for assessing NP+DI on its own terms but misleading as a recall claim. The honest base rate is 11.

Not on this page (yet): the parallel Django/Python real-world manifest (181 events, 4 exploited) — pulled out for now to land the point on a single ecosystem cleanly.

How we got here

  1. The 12-month synthetic backtest worked. NP+DI catches ~5 trigger dates per year on the Spring framework subset, 2-6 per year on each of Node/Django/Netty. Within any single 12-month window, almost every reasonable strategy looked perfect — small sample size hides differentiation.
  2. An AI safety-net scan closed the visible gaps. The CWE-misclassification cases (Ghostcat tagged CWE-269 instead of CWE-22; Tomcat partial PUT tagged CWE-44+502; ActiveMQ Jolokia tagged CWE-20 instead of CWE-94) get caught by an automated CWE-validation pass on NP-but-not-DI advisories. NP+DI + AI scan reaches 8/11 = 73% on real-world Java without filtering.
  3. We extended the backtest to a real 7-year window across the full real-world Java manifest to test whether the 12-month "perfect catch" framing survives at scale. The combined picture changes meaningfully.
  4. An adversarial hacker persona, blind to actual exploitation outcomes, ranked vulnerabilities by attack value (S = drop everything, A = strong campaign candidate, B/C/D = lower priority). The hacker's discriminator was default-config × network-edge × primitive-direct — derived independently from CVE structure on disclosure day, not from CWE-set membership.
  5. Hacker S+A turned out to be both the most efficient (fewest patches per exploit caught) and safest (fewest missed exploits) review methodology at 7-year scale. 10 of 11 actual exploits caught (91% recall) at 47 patches over 7 years — a meaningful improvement on every alternative.

The synthesis

The 12-month story (Table 1) holds on its own terms but doesn't differentiate. One exploit in window (ActiveMQ Jolokia). NP+DI misses it (CWE-20 not in DI), patch-criticals misses it (HIGH not CRITICAL), every other strategy catches it. At single-window scale on a Java estate, every reasonable strategy looks roughly equivalent.
The 7-year story (Table 2) is what shows the operational ordering. Hacker S+A is the only single-strategy choice with 90%+ recall on actual exploitation, at 4.7x oracle overhead. NP+DI alone is worse than patch-criticals on both axes (45% recall vs 64%, 6.8x vs 5.9x). NP+DI + AI scan brings recall to 73% at the lowest single-strategy overhead (4.6x), but still misses the 3 deserialization/utility-library exploits (Text4Shell, SnakeYAML, ActiveMQ OpenWire). The union NP+DI ∪ Hacker S+A reaches 100% recall at 5.7x oracle.

Counter-argument worth holding

  • The hacker exercise is a methodology validation, not a deployable product. Running an LLM-based hacker persona on every CVE batch is expensive and prompt-design-sensitive. The operational artifact is an AI scan extended to encode the hacker's discriminator as programmatic checks (auth-missing CWE co-tag rule, package-role-aware NP for utility libraries, CWE-502 deserialization promotion on network-facing components). That extension would close the 3-event gap between current AI scan (8/11) and Hacker S+A (10/11).
  • 11 exploited events is small N. The Hacker 10/11 vs NP+DI 5/11 gap is meaningful but flip 1-2 boundary events and headline numbers shift ~10pp.
  • Right-censoring affects 2024-2026 events. If recent CVEs accumulate exploitation evidence by 2027-2028, every strategy's recall improves; relative ordering should hold but absolute numbers shift.
  • Java enterprise is the most-attacked surface in history. The exploitation density is unusually high. Repeating this exercise on Python (Django manifest, 181 events / 4 exploited) shows similar directional results but with smaller base rates and less differentiation among strategies — separate analysis, separate dataset.
