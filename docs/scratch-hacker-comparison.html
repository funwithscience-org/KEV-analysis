Production manifest (Spring 4-5.x, Spring Boot 2.x, log4j 1.x + 2.x, XStream, ActiveMQ, plus build/scripting deps) — 7-year backtest with floor-sweep effective recall

Scope: the actual deployed production manifest. 60 unique packages across 176 version-pinned entries — multi-version snapshot of an enterprise estate with real-world version sprawl. 223 critical/high CVEs over 7 years; 13 actually exploited.

Worth being explicit about what's in this manifest (so nobody misreads it as a "modern Spring Boot 3 only" view):
  • Spring core stack: Spring Framework 4.x AND 5.x (spring-web 4.1.6 through 5.3.31 — 13 versions; spring-beans 4.1.6 through 5.3.13 — 8 versions), Spring Boot 2.2-2.7 (actuator-autoconfigure), Spring Security 4.x and 5.x, Spring Integration, Spring WS
  • Web servers: Tomcat embed-core 8.5 and 9.0 (5 versions), Tomcat coyote 6.x
  • Logging — both log4j lines: log4j:log4j 1.2.6 / 1.2.15 / 1.2.17 (the long-EOL 1.x line still in many estates), AND org.apache.logging.log4j:log4j-core 2.8.2 (the Log4Shell-vulnerable 2.x line). Logback also present.
  • Messaging / brokers: ActiveMQ activemq-all 5.12.0 (all-in-one client+broker jar), Apache MINA, Apache SSHD, Hazelcast cluster wire protocol
  • SOAP / XML web services: Apache CXF (api + databinding-aegis), Spring WS, dom4j 1.x and 2.x, Apache XMLBeans, Mulesoft variants
  • Deserialization gadget classics: commons-collections 3.2.x (the original ysoserial gadget chain), XStream 1.4.7 / 1.4.10 / 1.4.11.1, Jackson 1.x (jackson-mapper-asl) and Jackson 2.x (databind 2.3.1 through 2.9.8 — 7 versions), SnakeYAML 1.3 through 1.33 (9 versions), Apache BCEL 6.3
  • Scripting runtimes: Groovy 2.4.x, JRuby 1.7 + 9.2, Jython 2.7 — all RCE-capable when fed untrusted input
  • Build tooling and supporting libraries: Maven core/settings/shared-utils/surefire, plexus-archiver/utils, Apache Ivy, WireMock, Springfox Swagger UI, jxpath, woodstox-core, xmlunit-core
This represents a 2018-2026 production estate with real version sprawl — different applications in the estate are running different versions of the same package (jackson-databind has 7 versions, spring-web has 13, snakeyaml has 9). That diversity is the operational reality the analysis is testing against.

Effective recall — when "floor sweep" rescues a direct miss

The naive recall metric counts only direct catches. Per the build-mechanics analysis, every emergency rebuild triggered for one CVE applies the latest version of every dependency in the manifest. So a long-TTE exploit gets effectively patched by floor sweep if any other filter trigger fires during that window. The exposure ceiling for any specific exploit is "until the next filter trigger fires," not "until the filter catches this specific event." (See also two operational modes.)

On this manifest, NP+DI + AI scan fires 49 times across 7 years — roughly once every 50 days. That cadence is dense enough that any long-TTE event easily spans at least one other trigger. Of the 4 events NP+DI + AI scan misses directly:

  • CVE-2022-42889 Text4Shell (Oct 2022, ~437d TTE to MSF module) — Floor swept many times over: NP+DI+AI fired in Q4 2022 (2x), Q1 2023 (1), Q3 2023 (1), Q4 2023 (1).
  • CVE-2022-1471 SnakeYAML (Dec 2022, ~297d TTE) — Floor swept by NP+DI+AI fires in Q1 2023, Q3 2023, Q4 2023, Q1 2024.
  • CVE-2017-12615 Tomcat PUT 2017 (Oct 2018 OSV-publish, KEV 2022-03-25 = 1255d) — A 3.5-year window in which NP+DI+AI fired ~25 times. Effectively-caught many times over.
  • CVE-2017-12617 Tomcat upload 2017 (May 2022 OSV-publish, KEV 2022-03-25 — KEV actually predates OSV publish here, weird) — Caught via standard Tomcat update cycle.

So NP+DI + AI scan effectively catches 13/13 = 100% on this manifest when floor sweep is accounted for. The 2 "unrecoverable" events on direct catch (Tomcat upload pair) get patched anyway via routine Tomcat update cycles forced by other Tomcat triggers. Hacker S+A also reaches near-100% effective recall, with the same 1 caveat as before (CGI Win in 2019 with no other hacker triggers in 64-day window).

The operational read. Direct-catch comparison: comparator 9/13, hacker 9/13 (different mix). Effective-recall-via-floor-sweep: both reach near-100% because the manifest's trigger cadence is dense enough (one trigger every ~50 days) that long-TTE blind-spot events are mathematically certain to be floor-swept. Adding the union doesn't improve effective recall meaningfully. The hacker S+A advantage of "fewer patches per catch" (2.2x oracle vs 5.4x) is real but should be read with the * caveat about hacker tier coverage on the supplementary packages.

The synthesis

Direct catch: comparator and hacker tie at 9/13 = 69% on this manifest. They catch different events. NP+DI+AI catches the famous-bug cluster (Spring4Shell, Log4j x2, XStream RCE, Ghostcat-via-AI, Tomcat-PUT-via-AI, Jolokia-via-AI) plus the 1990s-era XStream cmd injection. Hacker catches the deser/utility cluster (Text4Shell, SnakeYAML) plus the Spring4Shell/Log4j/Ghostcat/Tomcat-PUT/Jolokia overlap. Neither catches the 2 Tomcat HTTP PUT file upload bugs from 2017 (CWE-434, deliberately out of NP+DI scope and hacker downgrades narrow file-upload primitives).
Effective recall via floor sweep: both reach near-100%. NP+DI+AI's 4 direct misses (Text4Shell 437d TTE, SnakeYAML 297d TTE, both 2017 Tomcat uploads with multi-year exploitation evidence accumulation) are all rescued by other NP+DI+AI triggers in their windows. Hacker S+A's 4 direct misses (the same Tomcat CGI Win + 3 events not in the original ranking input) similarly get rescued by routine Tomcat / XStream update cycles.

Where attack surface concentrates vs where it doesn't

  • Web servers + deserialization libraries dominate the exploited set — 11 of 13 exploits hit Tomcat (5), XStream (2), log4j-core (2), Spring (1), ActiveMQ broker (1). The remaining 2 are utility-library evaluator bugs (commons-text Text4Shell, SnakeYAML).
  • Build tooling adds patch volume but zero exploits. The Maven / plexus / ivy / surefire packages contribute 9 critical/high CVEs over 7 years (path traversal in plexus-utils, command injection in plexus-utils, dep-confusion in maven-core, XXE in ivy, etc.) but none have any KEV / Metasploit / ExploitDB entry. Both NP+DI+AI scan and Hacker S+A correctly downgrade these (build tools aren't network-edge production runtime), so neither strategy spends patch budget here. This is a clean validation that the discriminators aren't wasting effort on the build cluster.
  • Scripting runtimes (Groovy / JRuby / Jython) add ~12 more CVEs, also zero exploited in this window. Same pattern: real bugs, not network-edge, not actually weaponized in production-application contexts.
  • The 2 "unfixable" Tomcat CWE-434 events from 2017 are honest limit cases. File upload bugs (CWE-434) are deliberately out of NP+DI scope; the hacker discriminator B/C-tiers file uploads on the narrow-precondition rule. Only patch-everything catches them.
  • NP+DI + AI scan tops out at 9/13 direct because two structural blind spots persist: (1) deser-RCE on utility libraries (commons-text, SnakeYAML role=OTHER); (2) Tomcat file upload (CWE-434 excluded). Floor sweep covers both blind spots given the trigger cadence.

Counter-argument worth holding

  • Multi-version manifest semantics. The manifest has multiple versions of the same package (jackson-databind 2.3.1 through 2.9.8, Tomcat 8.5.23 through 9.0.91). A CVE that affects only a subset of versions still counts in this analysis because we're not doing version-range intersection. Real-world recall might be higher (because some CVEs only affect versions you don't run) or the same (because you DO run the affected version somewhere in your estate).
  • Hacker tier coverage. The 9/13 hacker direct catch and 20-patch flag count both depend on the original ranking input. A fresh hacker run would likely shift these slightly — probably catch the same 9 (the discriminator-clean ones) but flag more events on the supplementary packages.
  • Build vs runtime ambiguity. The manifest mixes runtime deps (spring-web, tomcat-embed-core, log4j-core) with build-time deps (maven-core, plexus-archiver, surefire-*). A runtime-only filter would have a smaller universe but probably similar exploit count. A build-time exploitation would be a different threat model.
