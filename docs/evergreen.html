Does Evergreening Change the Answer?

If you aggressively adopt new major versions of your platform, OS, and libraries as they ship — do you avoid vulnerabilities, or just trade old ones for new ones?

The Short Answer

It depends on the ecosystem — and the answer isn't what you'd expect.
Across Java, Node.js, RHEL, and Windows Server — evergreening doesn't consistently reduce CVE exposure. In some cases (Java 11) it eliminates confirmed exploitation. In others (Node.js 18+) it adds attack surface. The consistent finding: the runtime/platform generates CVEs that nobody exploits; the library/ecosystem layer is where confirmed damage happens.

Our periodicity analysis backtested a real-world Java runtime manifest — 58 packages covering Spring Framework, Spring Boot, Spring Security, Tomcat, jackson-databind, log4j-core, XStream, Apache CXF, MINA, SSHD, Hazelcast, ActiveMQ and the rest of a typical enterprise Spring Boot deployment — over 7+ years (2018–2026). That window contains 194 Critical/High CVEs across those packages: 40 NP+DI raw events (the structure test's emergency-rebuild flag), 43 NP+DI+DQ after the data-quality re-validation rescues 3 more, and 11 actually-exploited events with KEV / Metasploit / ExploitDB confirmation.

The evergreening question is operationally meaningful for the events that warrant emergency response — i.e. the NP+DI raw set (40 events). So we apply the per-CVE generation classification primarily to those 40, with the wider 194 reported as breakdown context. The manifest was built on the "old generation": Java 8, Spring 5, Spring Boot 2.x, Tomcat 8.5/9.

The evergreen alternative: upgrade to Java 11 (2018) → 17 (2021) → 21 (2023), Spring 6 (2022), Spring Boot 3 (2022), Tomcat 10 (2021) as each became available. For each CVE, we ask two questions: (1) does the new generation avoid the vulnerability? and (2) can you get a free patch?

Avoided: empty cohort on the canonical manifest

The canonical 58-package runtime manifest contains zero NP+DI raw events that evergreening would have avoided. The prior version of this page cited two — CVE-2019-10172 (jackson-mapper-asl, Jackson 1.x) and CVE-2019-17495 (Swagger UI via Springfox) — both of which arose from dead dependencies that the new-generation Spring Boot stack drops entirely. Neither package is in the canonical manifest because that manifest is built around the live runtime portfolio (jackson-databind 2.x, springdoc-openapi), not legacy abandoned deps. So the prior 2/2 framing was correct on its narrower scope but doesn't survive the reconciliation: when you backtest only the dependencies a real Spring Boot app actually ships with today, there are no "old-gen-only" emergency events for evergreening to rescue you from.

This is a directional shift from the prior page narrative, and it's worth being honest about: the "evergreening avoids 2" claim relied on counting fire-drills against dead packages a modern team wouldn't be running anyway. On a manifest reflective of current runtime reality, that lever returns nothing. The dependency-pruning insight still holds — drop dead deps — but it's a separate lever from generation shift.

The 2 events that evergreening made worse (Spring4Shell, ActiveMQ Jolokia) are both confirmed exploited with KEV entries and Metasploit modules. So the cross-tab is now stark: 0 avoided / 2 worse, both worse cases exploited. Evergreening's net effect on the in-scope cohort is unambiguously negative.

Both generations affected: 25 of the 40 NP+DI raw events

The plurality of NP+DI raw events affected both old and new generations simultaneously — the generation doesn't determine whether you have the bug. For these, evergreening doesn't help; patch currency on whichever generation you're running does. The patterns:

Tomcat (7 of 7 NP+DI events): Every Tomcat NP+DI event in the canonical manifest — CLIENT_CERT auth bypass, JsonAccessLogValve escaping, relative path traversal, header smuggling (CVE-2023-46589), JsonErrorReportValve escaping, Content-Length validation (CVE-2022-42252), Windows CGI injection (CVE-2019-0232) — affected all generations (8.5, 9, 10.1, 11) that existed at disclosure. Tomcat's core request-handling code is shared across generations. Fixed by minor patch, not generation upgrade.

Spring Security (8 of 8 NP+DI events): Authorization-bypass and broken-auth events affecting both 5.x and 6.x. The annotation-detection bypass (CVE-2025-41248), method-security private-method bypass (CVE-2025-41232), AuthenticatedVoter mispass (CVE-2024-22257), forward/include dispatcher bypass (CVE-2022-31692), RegexRequestMatcher bypass (CVE-2022-22978), and the older 5.x DoS (CVE-2021-22119), 4.x/5.x password handler (CVE-2019-11272), and OAuth (CVE-2018-15801) all affect the same code paths in both generations.

Spring Framework (2 of 3 NP+DI events, no_difference): The path-traversal pair CVE-2024-38816 and CVE-2024-38819 both affect Spring 5.3 AND 6.x. Free patch only on 6.x — but the bug is shared. Generation shift doesn't avoid the CVE; it preserves free-patch access (paywalled pattern, see below).

Apache CXF (3 of 3 NP+DI events): SSRF in MTOM (CVE-2022-46364), OAuth2 DoS (CVE-2021-22696), OIDC session hijack (CVE-2019-12419) — all in shared code across the 3.x major lines. Patch on whichever 3.x line you're on.

Log4j (2 of 2 NP+DI events): Log4Shell (CVE-2021-44228) and the incomplete fix (CVE-2021-45046) affected log4j-core 2.x regardless of what Spring/Java version you run. Spring Boot defaults to Logback; if you opted into log4j2, both Spring Boot 2 and 3 are equally affected.

Thymeleaf (2 of 2 NP+DI events): Expression-scope and unauthorized-expression bugs (CVE-2026-40477, CVE-2026-40478) affect thymeleaf 3.1.x regardless of whether you bind via thymeleaf-spring5 or thymeleaf-spring6.

ActiveMQ (1 of 1 NP+DI event): CVE-2019-0222 — OpenWire deserialization on the 5.15.x line. The Jolokia "made-worse" event (CVE-2026-34197) is a separate NP-only DQ-rescue case, not in the NP+DI raw 25.

Irrelevant: 14 of the 40 NP+DI raw events have no generation axis

For these, the package version is selected independently of Spring/Java generation — evergreening Spring or Java doesn't change which version you ship. The right lever is dependency-pruning or patch currency on the dep itself.

XStream (6 NP+DI events): Application-level dependency. All 6 events would only have been mitigated by dropping XStream entirely or pinning to ≥ 1.4.18 (which switched from denylist to allowlist). The XStream version isn't tied to Spring or Java. CVE-2021-39144 is the headline — KEV-listed, exploited via VMware Cloud Foundation.

jackson-databind (5 NP+DI events): Same library across Spring Boot 2 and 3 — CVE applies by jackson version, not by Spring/Java generation. The polymorphic-deserialization gadget chains (CVE-2020-24616, CVE-2020-25649, CVE-2019-14379, CVE-2018-14721, CVE-2018-14720) are mitigated by patch currency on jackson, not generation shift.

Hazelcast (2 NP+DI events): Standalone runtime dep. Its 5.x line is selected independently of Spring/Java. SQLi in CSV connector (CVE-2023-45860) and XXE in config parser (CVE-2022-0265) require pinning Hazelcast itself.

Apache MINA (1 NP+DI event): CVE-2024-52046 — deserialization RCE in the networking framework. App-level dep; pin or replace.

Does evergreening change the NP+DI answer?

No. NP+DI fires on CWE pattern + network-parser classification, not library version. Across the 40 NP+DI raw events: 1 makes-worse, 25 no-difference, 14 irrelevant, 0 avoided. You'd rebuild the same number of times on either generation. The filter is generation-agnostic because the vulnerability classes it targets (injection, parser abuse, auth bypass) exist in every generation.

The 7 paywalled CVEs: Spring 5.x after EOL

Seven of the ~31 "both generations affected" CVEs have a twist: they affect Spring 5.x and 6.x equally, but free patches only exist on 6.x. After Spring 5.3's OSS end-of-life (Aug 2024), Broadcom backports fixes to 5.3.x — but behind a commercial paywall. This is where evergreening matters: not because it avoids the vulnerability, but because it preserves your access to free patches.

To be clear: every one of these CVEs affects both Spring 5.x and 6.x. An evergreener on Spring 6 has the same bugs — the difference is purely that they can get a free OSS patch. These all sit in the "no difference" bucket of the canonical NP+DI raw set (CVE-2024-38816 and CVE-2024-38819 are explicitly there; the others fall outside the strict NP+DI raw filter but exhibit the same shared-bug-with-paywalled-fix pattern). The gap is about patch access, not vulnerability avoidance. Broadcom held the commercial-only line even for the CVSS 9.1 auth bypass — no EternalBlue-style free exception.

But none of them are being exploited. Zero KEV entries, zero Metasploit modules, zero ExploitDB entries across all 7 CVEs. The path traversals require functional web framework endpoints (a minority of Spring apps). The auth bypass requires WebFlux + static resources + specific security config. The actuator bypass requires CloudFoundry deployment. The conditions are narrow enough that despite high CVSS scores and unpatched target populations, attackers haven't shown interest.

This is consistent with the pattern already visible in the canonical NP+DI backtest: the only events evergreening would have avoided (the prior page's jackson-mapper-asl + Springfox examples) aren't in the canonical manifest at all because no modern Spring Boot app ships them. Meanwhile the 2 it made worse (Spring4Shell, ActiveMQ Jolokia) were both confirmed exploited. High CVSS + unpatched ≠ exploitation. The lifecycle debt is accumulating, but so far it's been a difference without a distinction.

The open question: does this stay true? The post-EOL window is only ~18 months old. The RHEL 6 ELS story took years to develop a meaningful patch gap. And these are NP+DI-class vulnerabilities on one of the most widely deployed web frameworks — the kind of target that could become attractive if the holdout population persists long enough. But the current evidence says: you can see the lifecycle risk accumulating on paper, and it's not translating to exploitation in practice.

The Java 11 discontinuity: less than it appears

All 13 Java CVEs in CISA KEV (CVE-2010-0840 through CVE-2016-3427) target the applet/Deployment sandbox — browser-side drive-by attacks where visiting a malicious webpage exploits the Java browser plugin. Java 11 removed this subsystem. But this was never a server risk. Server workloads don't run browsers, don't load applets, and never invoked the Deployment code path. The Deployment subsystem was dead weight on servers regardless of Java version.

These were browser plugin exploits. Browsers had already blocked Java plugins by 2015 (Chrome 45) and 2017 (Firefox 52). Java 11 (Sep 2018) formalized the removal of an attack surface that was already dead in practice. For server workloads, these CVEs were irrelevant at every Java version.

Patching model: no paywall trap

Unlike Spring under Broadcom, the JVM patching story has no paywall trap. Oracle patches all supported versions simultaneously in each quarterly CPU (like Windows Patch Tuesday, unlike RHEL). And critically, the OpenJDK ecosystem provides free backports:

Java versionOracle free updatesFree backport sourceStatus
Java 8Ended Jan 2019Red Hat (OpenJDK 8u), Adoptium, Amazon CorrettoActive — free patches within days of Oracle CPU
Java 11Ended Sep 2023Red Hat (OpenJDK 11u), Adoptium, Amazon CorrettoActive — same
Java 17Through Sep 2026Oracle + communityActive
Java 21Through Sep 2028Oracle + communityActive

Red Hat took over maintenance of OpenJDK 8u and 11u. Adoptium/Temurin builds follow within days of each Oracle CPU. There is no Broadcom-style commercial gate — Java 8 users on Adoptium get the same security patches as Java 21 users, for free, on roughly the same schedule.

The JVM evergreening story in one sentence: For server workloads, the JVM has zero confirmed exploitation at any version — the 13 KEV entries were all desktop/applet exploits irrelevant to servers. The curve is flat: same CVEs, same patches, same day. The JVM doesn't have a lifecycle trap because the open-source ecosystem provides free backports indefinitely. Evergreening from Java 8→11 removes dead code (Deployment, CORBA) that was never a server risk. Going further (11→17+) trades old-version-only CVEs for new-version-only CVEs at roughly the same rate, with Psychic Signatures (Java 15-18 only) being the most dangerous CVE in the dataset — and one that only existed because of the upgrade.

Zero modern exploitation

Perhaps the most striking finding: no Java SE CVE from 2017 onward appears in CISA KEV. The JVM itself hasn't been confirmed exploited in the wild in nearly a decade. All modern Java-ecosystem exploitation happens at the library layer (Log4Shell) or application server layer (WebLogic deserialization, 11 KEV entries). The platform is generating CVEs — 28-37 per year — but attackers aren't using them. This is consistent with the broader pattern: CVSS score alone doesn't predict exploitation.

The obvious explanation is "new subsystems" — user namespaces, eBPF, io_uring, nf_tables — that don't exist in older kernels. That's part of the story, but the KEV data reveals a bigger one: new code in old subsystems.

Where the bugs were introduced — by RHEL era: Of 14 unique kernel CVEs exploited on RHEL 8 or 9, 79% (11 of 14) were introduced before RHEL 8's kernel (4.18) — mostly in the RHEL 7 era (3.10–4.17). The real argument: lifecycle coverage

The case for RHEL evergreening isn't fewer CVEs — it's getting patches at all.

RHEL 6 went EOL November 2020. After that, without paid ELS ($400+/server/year), you get zero patches. With ELS, you get 2–3 advisories/year vs. 99+ on RHEL 8. And patch delivery is dramatically slower: CVE-2021-22555 (netfilter write) was patched on RHEL 8 within 34 days; the RHEL 6 ELS patch arrived 4+ years later.

During RHEL 6's full support phase (pre-Nov 2020), patch delivery was roughly comparable. The evergreening benefit is entirely about staying inside the support window where Red Hat actively patches everything.

The CIQ backporting caveat

CIQ (Rocky Linux) published research showing Red Hat's backporting rate declines sharply as a RHEL version ages — even during its supported lifecycle. RHEL 8.8 kernel commits showed decreasing backported fixes starting November 2021. Each new RHEL major version resets the backporting clock, but the same decay pattern repeats. Running the latest stable upstream kernel would be ideal, but vendor-frozen kernels are the enterprise reality.

The real evergreening argument: exploit mitigations

The CVE count and patch speed data suggest evergreening doesn't help much tactically. But there's a macro argument that's harder to quantify and probably more important: newer kernels make exploitation dramatically harder, even when the vulnerability exists. This is the Land & Expand framing — the OS determines how bad things get after the initial compromise.

RHEL 6 has approximately zero post-exploitation barriers. You're safe from nf_tables bugs on RHEL 6 the same way a horse is safe from flat tires — but when a bug does hit old code that exists in 2.6.32, the attacker's path to full compromise is a straight line.

Windows Server has the same story: Secured Core (HVCI, VBS, DMA protection) on Server 2022+ doesn't reduce CVE count but makes exploitation significantly harder. The mitigations are the reason to evergreen, not the patch math.

Express: not the Spring equivalent

Express 4→5 is not a Spring 5→6-style boundary. Express has had only 4 CVEs in its entire history (1 High). The npm ecosystem: where the risk actually lives

The Node.js evergreen story can't be told without the npm supply chain, which has no equivalent in the Java world at this scale.

lodash — the XStream of npm:

~46M weekly downloads, effectively unmaintained since 2021 (last meaningful release 4.17.21). Chronic prototype pollution: CVE-2018-16487, CVE-2019-10744 (Critical 9.1), CVE-2020-8203 (in CISA KEV), CVE-2021-23337, CVE-2026-2950. One maintainer. Unlike XStream, there's no architectural fix coming — the prototype pollution pattern is inherent to lodash's design of deep object manipulation.

vm2 — declared unfixable:

Sandbox escape library ultimately deprecated entirely after repeated Critical CVEs (CVE-2023-37466, CVE-2023-37903, CVE-2026-22709, all CVSS 9.8). The maintainers concluded the sandbox approach was architecturally unsound — the XStream denylist admission, but for JavaScript sandboxing.

Supply chain attacks dominate traditional CVEs:

event-stream (2018, social engineering → crypto theft), ua-parser-js (2021, account compromise → cryptominer), colors/faker.js (2022, maintainer sabotage), the September 2025 npm worm (chalk + debug compromised, 2.6B weekly downloads, self-replicating), axios (March 2026, lead maintainer credentials compromised). Over 99% of open-source malware now targets npm. Average npm project: 79 transitive dependencies.

Post-EOL: commercial-only, no community safety net

Unlike Java — where Red Hat, Amazon Corretto, and Eclipse Adoptium provide free extended LTS builds for Java 8 and 11 indefinitely — Node.js EOL support is commercial only (HeroDevs, NodeSource, TuxCare). Each LTS gets 30 months total (vs. Java's 8+ years of free community support). The Node.js project started publishing CVEs for EOL versions themselves in 2025 (CVE-2025-23087 for all Node ≤17) to flag the risk.

The evergreen pressure is stronger in Node.js than anywhere else we've examined. Shorter support windows (30 months vs. 8+ years for Java, 10 years for RHEL, 10 years for Windows Server), no free community backports after EOL, and the project willing to kill versions early when upstream dependencies (OpenSSL) go EOL. But the payoff for evergreening is worse — newer versions have more CVEs (new features = new bugs), the runtime isn't what gets exploited anyway, and the real risk (npm supply chain) is completely version-independent.

Synthesis: What Actually Reduces CVE Exposure?

Across all five layers — Java libraries, JVM platform, RHEL, Windows Server, Node.js — a pattern emerges, though with important variation:

StrategyReduces CVE count?Reduces exploitation risk?Mechanism
Evergreening (new major versions)No — equal or moreDepends on layerKernel: best case — 10 of 24 KEV entries (42%) exist only in old-gen code, vs 3 (12%) new-gen-only. Java libraries: unambiguously worse — on the canonical 58-pkg manifest, the old-gen-only cohort is empty (0 events); the new-gen-only cohort has 2 confirmed exploited (Spring4Shell, ActiveMQ Jolokia). The tactical answer: If you're trying to reduce how often you drop everything and rebuild, evergreening is not the lever. On the canonical 40-event NP+DI raw set, patch currency within your current generation (staying on the latest minor release) handles 25 no-difference events. Dependency pruning (do you actually need XStream?) handles 6 of the 14 irrelevant-cohort events. And NP+DI tells you which of the remaining patches actually require an emergency rebuild vs. riding the normal cycle.

Evergreening is still the right long-term strategy — you need to stay inside the support window, and the exploitation mitigations are real. But it's a strategic decision (lifecycle planning), not a tactical one (fewer fire drills). Spring 5.x post-EOL is accumulating NP+DI-class CVEs with no free patches — but none are being exploited yet, so the lifecycle debt remains theoretical.

The counterargument

There's a reasonable objection: evergreening keeps you close to upstream, which means patches arrive faster and the backporting decay problem (documented by CIQ for RHEL) is minimized. This is true — but it's a speed of patch delivery argument, not a fewer patches argument. And for the Java libraries, it's largely irrelevant: the fixes were version-of-the-library patches, not platform backports. Log4j 2.15.0 shipped the same day for everyone regardless of what JDK or Spring version they ran.

The strongest evergreening argument is probably transitive dependency freshness: Spring Boot 3's dependency management BOM pulls newer versions of its transitive dependencies, which means you're less likely to be running a 3-year-old version of a library with known vulns. But this is again dependency management — you can achieve the same thing with Dependabot or Renovate on any generation.
