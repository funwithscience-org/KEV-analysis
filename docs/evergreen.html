Does Evergreening Change the Answer?

If you aggressively adopt new major versions of your platform, OS, and libraries as they ship — do you avoid vulnerabilities, or just trade old ones for new ones?

The Short Answer

Evergreening shifts the attack surface.
It doesn't shrink it.
Across Java libraries, RHEL kernels, and Windows Server — newer versions consistently have equal or more CVEs to patch, not fewer. The case for staying current is lifecycle coverage and exploitation mitigations, not a smaller patch burden.

1. Java Libraries — The Real-World Manifest

Our periodicity analysis backtested a real-world Java manifest (Spring Boot app, ~40 dependencies) over 8 years (2018–2026), producing 25 NP+DI trigger CVEs plus 3 filter misses. The manifest was built on the "old generation": Java 8, Spring 5, Spring Boot 1.x, Tomcat 9.

The evergreen alternative: upgrade to Java 11 (2018) → 17 (2021) → 21 (2023), Spring 6 (2022), Spring Boot 3 (2022), Tomcat 10 (2021) as each became available. Would any of the 28 CVEs have been avoided?

Spring4Shell is the headliner. The single most impactful exploited CVE in this dataset was one where the "old" stack (Java 8) was immune and the "modern" stack (JDK 9+) was vulnerable. An evergreener would have had to emergency-patch; the team that stayed on Java 8 could have ignored it entirely.

No difference: 24 of 28 CVEs

The vast majority affected all supported major versions simultaneously. The patterns:

Tomcat (7 CVEs): Every Tomcat CVE — Ghostcat, request smuggling, PUT deserialization, path traversal — affected all generations (8.5, 9, 10, 10.1, 11) equally. Tomcat's core request-handling code is shared across generations. Fixed by minor patch, not generation upgrade.

XStream (6 CVEs): XStream is an application-level dependency. Its version is unrelated to Spring/Java generation. All 6 CVEs would only have been avoided by dropping XStream entirely — dependency pruning, not evergreening.

Spring Framework (3 CVEs): CVE-2018-1275/1270 (SpEL RCE) affected Spring 4.x AND 5.x. CVE-2023-20863 (SpEL injection) affected Spring 5 AND Spring 6. Both generations vulnerable, fixed by patch release.

Log4j (2 CVEs): Log4Shell affected Log4j 2.0–2.14.1 regardless of what Spring/Java version you run. Spring Boot defaults to Logback; if you opted into Log4j2, both Spring Boot 2 and 3 are equally affected.

Everything else (dom4j, jackson-databind, Hazelcast, Apache CXF, Apache MINA): All generations affected. These are transitive or application-level dependencies whose vulnerability is version-of-the-library specific, not platform-generation specific.

Does evergreening change the NP+DI answer?

No. NP+DI fires on CWE pattern + network parser classification, not library version. The 2 avoided CVEs (jackson-mapper-asl, Swagger UI) were both NP+DI triggers — so the filter would have fired either way. The 2 "made worse" CVEs were also NP+DI triggers. You'd rebuild the same number of times on either generation. The filter is generation-agnostic because the vulnerability classes it targets (injection, parser abuse, auth bypass) exist in every generation.

2. RHEL — Newer Kernels, More Attack Surface

The "current" RHEL version consistently receives the most advisories, not the fewest. This partly reflects that Red Hat patches all severities on current versions but only Critical/Important on older ones — but the sheer volume gap (125 vs 2) is too large to be explained by policy alone.

Why newer kernels have more CVEs

Each RHEL major version adds kernel subsystems that become fresh attack surface: user namespaces, eBPF, io_uring, OverlayFS, nf_tables improvements, cgroup v2, fs_context. These are valuable security infrastructure (namespaces enable containers, eBPF enables monitoring), but they carry their own bugs.

The real argument: lifecycle coverage

The case for RHEL evergreening isn't fewer CVEs — it's getting patches at all.

RHEL 6 went EOL November 2020. After that, without paid ELS ($400+/server/year), you get zero patches. With ELS, you get 2–3 advisories/year vs. 99+ on RHEL 8. And patch delivery is dramatically slower: CVE-2021-22555 (netfilter write) was patched on RHEL 8 within 34 days; the RHEL 6 ELS patch arrived 4+ years later.

During RHEL 6's full support phase (pre-Nov 2020), patch delivery was roughly comparable. The evergreening benefit is entirely about staying inside the support window where Red Hat actively patches everything.

The CIQ backporting caveat

CIQ (Rocky Linux) published research showing Red Hat's backporting rate declines sharply as a RHEL version ages — even during its supported lifecycle. RHEL 8.8 kernel commits showed decreasing backported fixes starting November 2021. Each new RHEL major version resets the backporting clock, but the same decay pattern repeats. Running the latest stable upstream kernel would be ideal, but vendor-frozen kernels are the enterprise reality.

Server 2022 had 151 more CVEs to patch than Server 2016 in 2025 (25% more). Server 2025 had 177 more (29% more). This pattern held every single month — not once in 20+ months of data did Server 2016 have more Patch Tuesday CVEs than Server 2022.

~65% of CVEs hit every supported version. The remaining ~35% are almost entirely "newer versions only" — new subsystems (Brokering File System, Virtual Secure Mode, Azure Local, PrintWorkflowUserSvc) that older versions never had.

Across all of 2024, only 5 CVEs affected Server 2016 but not newer versions. In 2025, just 1 CVE. The "legacy-only" CVE category barely exists.

Notable CVEs: version doesn't matter

PrintNightmare (CVE-2021-34527): ALL versions. Patches released simultaneously. Core print spooler code is shared.

Zerologon (CVE-2020-1472, CVSS 10.0): ALL versions (2008 R2 through 2019). Netlogon protocol is shared infrastructure.

EternalBlue (CVE-2017-0144): The one genuine legacy example — SMBv1 is not installed by default on Server 2019+. But SMBv1 can also be disabled on Server 2016. This is a configuration choice, not a version-freshness issue.

The real argument: exploitation mitigations

The case for Windows Server evergreening isn't fewer CVEs — it's making CVEs harder to weaponize.

Server 2022+ ships with Secured Core (HVCI, DMA protection, VBS), TLS 1.0/1.1 disabled by default, and TPM 2.0 for firmware protection. These don't reduce the CVE count — they reduce exploitability. An attacker hitting a CVE on Server 2022 faces more hurdles than the same CVE on Server 2016.

And like RHEL, there's a lifecycle argument: Server 2016 mainstream support ended January 2022, extended support runs to January 2027. Staying current buys runway before you hit end-of-support.

4. Synthesis: What Actually Reduces CVE Exposure?

Across all three axes, the same pattern emerges:

StrategyReduces CVE count?Reduces exploitation risk?Mechanism
Evergreening (new major versions)No — equal or moreYesLifecycle coverage, exploitation mitigations, legacy protocol removal
Patch currency (latest minor/patch)YesYes26 of 28 Java CVEs fixed by point releases within existing major lines
Dependency pruning (drop unused libs)YesYesDropping XStream alone eliminates 6 of 28 CVEs regardless of generation
NP+DI filterYes (80% reduction)YesWorks identically across generations — fires on CVE structure, not version
The tactical answer: If you're trying to reduce how often you drop everything and rebuild, evergreening is not the lever. Patch currency within your current generation (staying on the latest minor release) handles 24 of 28 CVEs. Dependency pruning (do you actually need XStream?) handles another 6. And NP+DI tells you which of the remaining patches actually require an emergency rebuild vs. riding the normal cycle.

Evergreening is still the right long-term strategy — you need to stay inside the support window, and the exploitation mitigations are real. But it's a strategic decision (lifecycle planning), not a tactical one (fewer fire drills).

The counterargument

There's a reasonable objection: evergreening keeps you close to upstream, which means patches arrive faster and the backporting decay problem (documented by CIQ for RHEL) is minimized. This is true — but it's a speed of patch delivery argument, not a fewer patches argument. And for the Java libraries, it's largely irrelevant: the fixes were version-of-the-library patches, not platform backports. Log4j 2.15.0 shipped the same day for everyone regardless of what JDK or Spring version they ran.

The strongest evergreening argument is probably transitive dependency freshness: Spring Boot 3's dependency management BOM pulls newer versions of its transitive dependencies, which means you're less likely to be running a 3-year-old version of a library with known vulns. But this is again dependency management — you can achieve the same thing with Dependabot or Renovate on any generation.
