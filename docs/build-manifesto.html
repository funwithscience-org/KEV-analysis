The Build Manifesto

Engineering principles for build process — quality through change, not control.

The argument, in one paragraph

We have been taught that the path to quality is deterministic control of every input. That equation is wrong, and the cost of believing it is enormous. The path to quality is dynamic, well-tested inputs producing immutable outputs. Determinism belongs at the deployment artifact, not at the build inputs. The "stable build = stable product" intuition that gates a snakeyaml 2.3 → 2.4 bump with the same review process as a feature release is producing applications that cannot be rebuilt, cannot be upgraded, and cannot be patched. This document is the case for the alternative.

The principles below are not a security document. They are aligned with practices documented in Software Engineering at Google and now mainstream across the high-functioning end of the industry. They stand on their engineering quality independently. The vulnerability-management benefit is a side effect — a significant one, and often the political impetus that justifies the change, but not the goal.

The cultural reframe this requires is real. The same review machinery that exists to protect the organization from a bad business-function release is being applied to a snakeyaml patch bump where the upstream maintainer has already done the work. That misallocation of caution — treating safe, well-tested external change with the same friction as risky, novel internal change — is the bug. The fix is not "less caution." The fix is to put the caution in the right place: tests gating the dynamic flow, not pins gating every reference.

The three principles

Principle 01

Build-time dependencies should be dynamic

Both libraries and base images. Resolved at build, not pinned at code-write time. The build is a fresh integration test against the current state of the world.

Principle 02

Deployment is where you want static structure

The artifact you ship is immutable and self-contained. Lockfiles, signed images, SBOMs capture exactly what got resolved. Rollback works because the deployed thing is a snapshot, not a recipe.

Principle 03

If you depend on a behavior, write a test for it

The Beyoncé rule. Behavioral dependencies expressed as tests, not as version pins. If there is no test, you do not actually depend on the behavior — you just happen to be running with it today.

Build dynamic, deploy static

The cultural confusion is mistaking "stable deployment" for "stable build inputs." They are different problems with opposite right answers.

Stable deployment is the goal. Byte-for-byte identical artifacts running in dev, staging, and prod. That property is enforced at the deployment boundary by lockfiles, signed images, content-addressed storage, and immutable infrastructure. Once a build resolves, the artifact is fixed forever. Rollback works because the deployed thing is a self-contained snapshot — the binary, the lockfile, the image digest — none of which change after the build commits to them.

Stable build inputs is the anti-pattern. Pinning every dependency in source so that "the build is reproducible" produces apps that can be rebuilt to a multi-year-old version of the world, against multi-year-old versions of every library, with multi-year-old security holes baked in. It looks like control. It is actually a slow-motion accumulation of debt that compounds quietly until the day you need to upgrade something and discover that nothing builds anymore.

The right model: build dynamic, freeze at the deployment boundary. Every build pulls fresh dependencies, pulls fresh base images, runs tests, resolves to a concrete versioned artifact, captures the resolved state in a lockfile, and ships that artifact immutably. Each build is a fresh integration test against the current state of the world. The deployed artifact is fully deterministic. The build inputs are not.

The reframe in one sentence. Pin at the deployment boundary, not at the source-code boundary. The deployed binary is a snapshot of resolved state; the source manifest declares intent.

This is the model used by every healthy software organization at scale — Google, Netflix, Stripe, Shopify, modern Spring/Spring Boot teams. It is not exotic. It is the table-stakes engineering practice that the industry converged on more than a decade ago. The reason organizations still operate the pinned-everything way is cultural inertia from an era when "reproducible build" meant "exact bytes from source," and the answer was to pin every input. The industry learned that lockfiles + immutable deployment artifacts give you the same property at the only place it actually matters — what's running — without the staleness cost.

The Beyoncé Rule

Google's Software Engineering at Google codifies a principle they call the Beyoncé Rule:

"If you liked it, then you should have put a test on it."

Named after the song Single Ladies. It is a one-line cultural framework with significant implications.

The rule says: if your code depends on some behavior — of a library, a system call, a runtime, an API contract — that dependency should be expressed as a test that fails when the behavior changes. If there is no test, you do not actually depend on the behavior. You just happen to be running with it today, and you are silently betting that no one upstream will change it.

The corollary is the cultural unlock: when a dependency upgrade "breaks" your code, the upgrade is not the bug. Your test gap is the bug. The dependency was free to change in ways you didn't notice because you didn't write down what mattered. The library author behaved correctly under the contract you wrote (the empty one). You inherited a brittle dependency.

This reframes the version-pinning argument completely. "We pinned snakeyaml at 2.3 because we depend on its behavior" is not an engineering position — it is a confession that the team did not write a test for the behavior they care about. The fix is not to pin harder. The fix is to write the test, then go dynamic.

In a Beyoncé-Rule-respecting org, the question "is this dependency upgrade safe?" is answered by running the test suite. Green means safe. Red means a behavior the team explicitly cared about has changed, and now needs human attention. Either the upgrade is taken or the test is updated. The version is not pinned defensively.

The implicit prerequisite. The Beyoncé rule presumes a healthy test culture. Organizations that have under-invested in tests cannot safely absorb dynamic builds — they have no signal to gate on. The honest sequence is: build the test culture first, then go dynamic. Trying to skip step one produces "we let everything through and pray," which is worse than where you started.

Concrete actions

Operationalizing the principles produces a specific set of moves. None of these are speculative; all are mainstream, well-tooled, and used at scale today. Actions 01–03 are conceptually unified — they're how you absorb the ecosystem's goodness into the work you're already doing. Action 04 is structurally different — it's a separate operational-readiness goal we'll track distinctly.

Continuous ecosystem absorption — Actions 01–03

These three are one idea expressed at three layers (images, framework BOMs, the long tail). The throughline: as you do business work — building features, fixing bugs, shipping releases — the ecosystem's improvements ride in alongside. Patches arrive in BAU because the build mechanism makes them arrive in BAU. This produces what's best described as productive chaos: the build inputs change underneath you, sometimes daily, and your test suite tells you whether your code still works against the current state of the world. If something breaks, that is your Beyoncé moment — the ecosystem just surfaced a behavior you depended on but didn't write down. Write the test, lock down the contract, move on. The chaos isn't a defect of the model; it's the model's mechanism for finding what you actually care about.

Action 01 — Pull in dynamic images at each build

Base images referenced by moving tags rather than fixed digests in source. FROM company/runtime:current or FROM company/jdk21:lts, not FROM company/runtime:v1.2.3. The image gets resolved at build time. The deployed container is built from a specific resolved digest, captured in the deployment manifest. Source declares "use the current runtime"; deployment captures "this exact runtime digest as of this build."

Benefit: kernel patches, base-OS package updates, JDK security updates, runtime-library fixes — they all flow into your applications on the next rebuild without anyone touching application code. Without this, every base-image bump becomes a coordinated PR sweep across every Dockerfile in the org.

Action 02 — Move declarative libraries to BOM + starter "get best" calls

Replace per-version pins in build files with a combination of starter dependencies (which carry their own resolved version sets — Spring Boot starters being the canonical example) and individual library references against a centrally-maintained BOM. The application's pom.xml stops naming versions for most dependencies; the BOM dictates them. When a CVE drops in any library the BOM tracks, the platform team cuts a new BOM revision and every consumer rebuilds against the new versions automatically.

This is the high-leverage move. One BOM revision propagates across the entire estate; the alternative is N-thousand bot-driven PRs each cycle. Same outcome, dramatically less mechanical work in steady state. See the Operational Model §3 for the detailed mechanics of platform BOMs and the hybrid platform-plus-DIY shape that mature portfolios actually run.

Action 03 — Enable auto-fix for remaining declarative dependencies

For the long tail of build files that haven't been migrated to a BOM (and for genuine one-off dependencies that don't belong in the platform), use Renovate-style functionality to bump pinned versions automatically. We will use Nexus's built-in version (Sonatype Lifecycle's Automated Pull Requests) rather than standalone Renovate so the policy lives alongside the artifact governance already in Nexus. Auto-merge gated by green CI, scoped to the security-tracking package set. The bot commits a literal version bump to the source manifest; what you see in source is what you get at build.

This handles the cases the BOM doesn't reach. The two mechanisms are complementary, not competitive: BOM for the high-volume coordinated dependencies that scale linearly across the estate; auto-fix for the long tail. Both paths absorb patches in BAU.

Operational-readiness goal — Action 04 (tracked separately)

Action 04 isn't part of the ecosystem-absorption story. It's a goal we track because it does two things the absorption layer doesn't:

  1. Technical mechanism for a firm imperative. When something goes sideways — security event, regulator request, vendor pulls a release, contract review — and the answer is "we need to push a fresh build of that app today," you need an actual mechanism that produces a build of that app today. Without action 04, the answer is "give us two weeks while we figure out what broke in the pipeline." With it, the answer is "running now."
  2. Separation of populations — chase vs. programmatic. This is the corollary to action 01: once continuous absorption is the default, the apps where it works invisibly are not the apps you spend human attention on. The apps where you have to chase someone — find an owner, debug a rotted toolchain, hand-craft a deployment — are a distinct population that deserves explicit tracking. Action 04 is the operational property that lets you draw that line. "Triggerable" sorts the estate into "we can mechanically push this if needed" and "we cannot." Both are useful answers; not knowing which is which is the failure mode.

Action 04 — Every build pipeline must be triggerable

The pipeline can be invoked end-to-end on demand — by a person, a bot, a security event, or a scheduled rebuild — without an archeology project. This is distinct from Cat 1's "auto-builds on every commit." A Cat 2 app doesn't build continuously, but its pipeline must still be in working order: anyone with appropriate auth can push a button, the pipeline runs, and a fresh artifact lands. If the pipeline only works when its original author runs it from their laptop with three undocumented environment variables, you don't have a triggerable build — you have a Cat 3 app wearing a Cat 2 fig leaf.

The discipline is exercising the trigger. A pipeline that hasn't been invoked in 18 months has almost certainly rotted — upstream base image deprecated, test runner version bumped past the syntax the suite uses, secret credentials expired, hosting moved. The mitigation is to rebuild the orphan estate on a schedule (weekly or monthly, whatever cadence catches drift before it becomes a project) and surface failures explicitly. The platform team typically owns this; the metric is "% of estate verified-rebuilding," not "% of estate with a build config." The two are very different numbers in practice. See Operational Model §7 for the detailed argument about how Cat 2's hidden cost is triggers rotting.

Action 04 is useful even when it doesn't trigger a full retest. "Trigger the pipeline weekly" doesn't have to mean "run the full regression suite weekly." A smoke trigger — invoke pipeline, build artifact, check it boots and the health endpoint responds — gets you most of the value at a fraction of the cost:
  • Pipeline-rot detection. You find out the build is broken before the imperative arrives, not while triaging it. Most rot is infrastructure (expired credentials, deprecated base image, registry endpoint moved) — a smoke trigger catches it; full regression isn't required to surface it.
  • Inventory accuracy. "Which apps can I actually push today?" becomes an empirical question with a current answer, not an aspirational one. The estate's verified-rebuilding count moves with the smoke-trigger cadence.
  • Build-artifact freshness. When the imperative comes, you have a recent successful build to push. Even without a full validation pass, "current binary that boots and responds to /health" is dramatically better than "binary from 18 months ago, with three known CVEs in dependencies, that may or may not assemble today."
  • Population separation. Apps that pass the smoke trigger go in the "programmatic" bucket; apps that fail go in the "chase" bucket. That sort is the input to where humans should spend attention.
The full-regression test cadence stays whatever it is for that app's release model. Smoke-trigger cadence is the cheaper, more frequent signal that keeps the operational-readiness number honest.

What this isn't

Honest constraints worth being explicit about before someone interprets this as a license to skip discipline:

This isn't "no version control on dependencies." The deployment artifact is fully version-pinned via lockfile, image digest, or SBOM. The cultural argument is about where the pin lives — at the deployment boundary, captured automatically, not in the source manifest, written by hand and frozen forever.

This isn't a license to skip testing. The opposite. Dynamic build inputs only work if the test suite is honest about what behavior the code depends on. Organizations that don't invest in tests cannot safely absorb dynamic builds — they have nothing to gate on. The implicit prerequisite of Principle 01 is Principle 03.

This isn't an argument against pinning critical dependencies. Some things — cryptographic libraries that need to match validated FIPS configurations, libraries with known forward-compatibility issues, legacy components in maintenance — should be pinned with documented justification. The argument is against pinning by default. Default to dynamic; pin where you have a documented reason; review the pinned set on a schedule and unpin where you can.

This isn't free. Dynamic builds require CI infrastructure that can keep up. The platform team has real ongoing work — maintaining the BOM, smoke-testing new revisions before publishing, running the auto-fix bot fleet, owning the rebuild-orphan-apps-on-a-schedule pipeline. The investment is real and ongoing. The payoff is that the work concentrates in one place that can be staffed properly, instead of being smeared across every team that owes its own upgrade ceremony.

Some compliance frames require source-to-binary reproducibility. Aerospace, certain medical contexts, certain financial regulatory frames have explicit requirements that the same source produces the same binary, audit-traceably, possibly with regulatory pre-approval of inputs. For these, dynamic input flows through a vetted gate — a frozen BOM revision, a signed base image set, an audited lockfile — but the principle still holds: determinism at the deployment artifact, not at source-write-time. The gate is added to the BOM publication process, not to every individual build manifest.

The vulnerability side-effect

This document is not about vulnerability management. The principles stand on their engineering quality independently. Catching integration drift early, keeping upgrade paths open, expressing behavioral dependencies as tests, avoiding code-museum syndrome — these are good engineering whether or not security is the motivator.

That said, the practical effect on security posture is dramatic.

An organization operating this way absorbs security patches in the normal flow of business. SnakeYAML ships a CVE on Tuesday; tomorrow's build pulls the patched version; tests pass; the deployment proceeds; nobody filed an emergency ticket. The vulnerability-response shape that requires multi-week emergency floor sweeps in pinned-everything orgs becomes invisible BAU in dynamic-build orgs. The same shift that makes engineering healthier makes security cheap. The Operational Model page documents the full mechanics of how this absorbs into Cat 1 / Cat 2 / Cat 3 estate maturity, including the platform-team smoke-test obligation that makes it work for the parts of the estate that don't have their own CI.

Take the impetus. The argument for adopting these principles often gets political traction because of an acute vulnerability event — a Log4Shell, a Spring4Shell, a CVE that lit up the inbox at 2 AM. That's fine. The principles are right regardless of what motivated the conversation, and the side-effect is real. Use the moment.
