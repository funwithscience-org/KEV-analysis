Cross-Framework Patch Periodicity Analysis

Three triage strategies — NP+DI, NP+DI + DQ, and a hacker discriminator (default-config × network-edge × primitive-direct, with auth-missing co-tag rule) — scored against the same Spring Boot, Node.js/Express, Django, and Netty sample stacks plus a 7-year production manifest. WAF-defensibility added as a fourth axis on the actually-exploited events.

Working document — April 26, 2026 • Data: OSV.dev, CISA KEV, Metasploit, ExploitDB, EPSS • 12-month window: Apr 2025 – Apr 2026 • 7-year window: 2018-Q4 – 2026-Q2

The Question

In patch management, the binary decision is "do we rebuild or not." CVSS doesn't matter much because when you patch, you install everything. The old question was "is there a critical?" — but there's always a critical, so the trigger fires constantly.

We're proposing a sharper filter: only trigger an emergency rebuild for direct injection vulnerabilities in network-parser libraries. Everything else rides the normal update process (whatever cadence the team already uses picks up these patches automatically).

But this only matters if the filter actually spaces out the rebuild events. If you're still getting a trigger every week, you haven't gained anything — you've just added complexity. So: what's the periodicity under each filter, across three different tech stacks?

What's new in this preview vs. the published periodicity page:

Three additions are layered into the same flow: (1) a hacker S+A discriminator sits alongside NP+DI as a parallel comparator on every chart and table — default-config × network-edge × primitive-direct with auth-missing co-tag rule. (2) The NP+DI + DQ safety net (already in EPSS section here) gets per-event detail across the 7-year backtest. (3) A WAF-defensibility axis (FRIENDLY / MEDIUM / HOSTILE) is added to the 13 actually-exploited events to surface where perimeter mitigation can buy time and where it cannot.

The filter works across ecosystems — without missing any exploited vulnerability. The original three stacks each produce 14 distinct all-C/H trigger dates — you'd rebuild roughly every 3 weeks under the old model. Under NP+DI (including auth bypass via input manipulation), Spring drops to 5 events (64%), Node.js to 2 (86%), Django to 6 (67%), Netty to 1 (93%). Across the 12-month synthetic stacks and the 7-year real manifest backtest, every CVE that was later confirmed exploited (CISA KEV or Metasploit) landed in the NP+DI bucket. The filter never told you to skip something that turned out to matter.

*The caveat: the filter logic is sound, but it's only as good as the CWE metadata it reads. The 7-year backtest found 3 exploited CVEs that the filter missed — all because upstream CWE tags were wrong (Ghostcat tagged CWE-269 instead of CWE-22, ActiveMQ tagged CWE-20 instead of CWE-94). The filter's answer was correct given its input; the input was bad. The fix is an DQ safety-net: automated CWE validation for network-parser advisories that don't match a DI pattern. That catches all 3 on day zero. Trust the filter, but verify the data it reads.

Periodicity Comparison

Why averages lie here. The NP+DI filter doesn't produce evenly spaced events. It produces silence punctuated by bursts. Spring had 5.5 months of silence, then two hits in a week. Node.js had 10 months of silence, then two hits in 29 days. Averaging obscures the real operational picture. The burst pattern is actually better than a steady cadence — it means the filter truly clears the noise floor, and when it does fire, you know to take it seriously.

The real value of the 12-month view is showing how much needless rebuild churn the all-C/H baseline produces. Spring/Node/Django each disclose roughly 14 critical/high CVEs per year on these manifests — that's a forced rebuild every 2-3 weeks if you treat severity as the trigger. Even Netty's tiny 7-package stack throws 3 per year. Under any of the discriminating filters (NP+DI, NP+DI+DQ, hacker S+A) the rebuild count drops to 1-7, depending on framework and policy — and the ones that get filtered out are DoS bugs, weak crypto config, info leaks, and deserialization that can't actually be reached. The strategies disagree on small membership-and-count details but they agree on the headline: the all-C/H rebuild treadmill is mostly noise. Scoring which discriminator is most accurate needs ground truth — the 12-month window has only 1 exploited CVE total (Django CVE-2025-64459, EDB-only), so the 7-year backtest is where actual scoring happens.
The hidden cost of all-C/H rebuilds isn't just the rebuild itself — it's what the churn crowds out. Teams that fight a fire every two weeks don't typically use that bandwidth to fix root causes. They don't get to "let's adopt a get-latest build model so the next CVE is a non-event." They don't get to climb from Cat 3 (long, governance-blocked rebuild path) to Cat 2 (regular cadence) on the Build Mechanics estate-maturity ladder. The natural reflex is "we should just fix things right" — but humans in firefighting mode don't behave that way; they triage, ship, and brace for the next alert. A noise-floor reduction from 14 forced rebuilds per year to 2-6 isn't just a workload win — it's the difference between a team that only does emergency rebuilds and a team that has time to invest in not having to.

What the Filter Caught (and Why It Was Right)

Spring Boot 5 events

Oct 27: Tomcat path traversal (CWE-23) — attacker sends crafted HTTP request, traverses outside web root. Classic direct injection.

Apr 9: Tomcat output encoding flaw (CWE-116) — improper encoding in JsonAccessLogValve.

Apr 15: Thymeleaf template injection ×2 (CWE-1336/917) — expression language injection via template processing. Textbook SSTI → RCE.

The Thymeleaf SSTI is exactly the vulnerability class that reaches KEV.

Node.js 2 events

Feb 26: Koa host header injection (CWE-74) — ctx.hostname trusts raw Host header without validation. Enables cache poisoning, password reset hijacking.

Mar 27: Handlebars code injection ×4 (CWE-94) — AST type confusion allows JavaScript execution via crafted templates. Four separate attack vectors in the same disclosure batch.

Template engine code injection — same vulnerability class as the Thymeleaf hit in Spring.

Django 4 events

Sep 8: Django SQL injection via column aliases (CWE-89).

Oct 1: Same pattern, different code path (CWE-89).

Nov 5: SQL injection via _connector kwarg in QuerySet/Q (CWE-89).

Feb 3: Two more Django SQL injection fixes (CWE-89).

Recurring ORM-level SQLi. Django kept needing patches for the same class of bug — the filter correctly flagged every one.

Netty 1 event

Mar 26: Netty HTTP request smuggling via chunked extension quoted-string (CWE-444) — malformed chunked encoding allows front-end/back-end desynchronization.

The only trigger was CWE-444 — the request smuggling class added to the DI set after external review. Without CWE-444, this stack would have had zero NP+DI triggers in 12 months.

What the Filter Skipped (and Why That's OK)

Across all three stacks, the filter let 34 C/H events ride the team's normal update process — whatever that looks like in practice. (Mileage varies by estate-maturity category: Cat 1 teams absorb these in BAU; Cat 2 teams pick them up on a regular cadence; Cat 3 teams pick them up reactively when something forces an update. The discriminator's job is to keep these out of emergency response, not to dictate which category they land in afterwards.) These fell into predictable categories:

Interesting edge cases the filter handled correctly:

Sequelize SQL injection (Node.js, CWE-89): Filtered out because Sequelize is non-NP — it's an ORM, not a network parser. ORM-level SQLi requires the application to pass unsanitized input through specific query patterns. It's a real bug but the attacker path is indirect.

lodash code injection (Node.js, CWE-94): Filtered out because lodash is non-NP. The _.template injection requires attacker-controlled template strings — extremely rare in production unless you're already compromised.

black path traversal (Django ecosystem, CWE-22): Filtered out because black is a code formatter — essentially local attack surface only.

The Tomcat HTTP PUT 2017 pair: a worked example of why "B-tier handled by supplements" works.

CVE-2017-12615 was disclosed in September 2017; CVE-2017-12617 in October 2017. Both require readonly=false on the Tomcat default servlet — non-default config. Neither hit CISA KEV until late 2021 / 2022 — roughly four years after disclosure. That four-year gap is the operational evidence the hacker's B-tier rating was correct.

In that window, any team running floor-sweep patches on adjacent Tomcat NP+DI / hacker-S+A triggers would have been carried to a current Tomcat version that no longer has the vulnerable file-upload behavior. The actual Tomcat-adjacent emergency rebuilds in those 4 years:

  • 2018 jackson-databind cluster (multiple Tomcat-fronted services patched)
  • 2019 Tomcat CGI command injection (CVE-2019-0232, KEV)
  • 2020 Ghostcat AJP file read (CVE-2020-1938, KEV)
  • 2021 Log4Shell-driven full rebuild (every Tomcat instance touched)
  • 2021 Tomcat smuggling (CVE-2021-25122, MSF)
  • 2022 Spring4Shell + Tomcat CLIENT_CERT

Six emergency rebuilds in the four-year window between disclosure and KEV. Any one of them would have absorbed the file-upload patches via floor-sweep. The hacker's B-tier was right: low immediate-exploitation risk, narrow precondition, supplements handle it on the normal cadence rather than emergency.

The third defense-in-depth layer: threat intel. Floor-sweep + cadence is what actually caught the 2017 pair. Threat intel didn't need to fire here — but it's the third layer of defense for a different future scenario where a B-tier event's exploitation accelerates faster than natural cadence would catch it (exploit lands in 30 days, not 4 years). In that case the watch list flags it and the event is promoted out of B. With all three layers in place — floor-sweep, cadence, and threat-intel monitoring — the model is never worse than EPSS: structure tests fire on disclosure day for the 11/13 the model directly catches, cadence + floor-sweep handle the long-latency tail, and threat intel covers the accelerated-exploitation case EPSS would otherwise be the only catch for.

The misreading to guard against: "B-tier means ignore." It doesn't. The model's claim is "B-tier handled by supplements" — and that depends on actually running the supplements. A team that drops get-newest builds, regular cadence, threat-intel monitoring, application metadata, and targeted WAFs doesn't get the B-tier-absorbed-by-supplements behavior; they get the B-tier-eventually-exploited behavior. The model is the structure tests and the supplementary controls together.

The Cross-Ecosystem Pattern

Four different stacks (Spring, Express, Django, Netty), three package ecosystems (Maven, npm, PyPI), different dependency philosophies — but the same structural result:

1. The all-C/H noise floor is consistent. The traditional three stacks each hit 14 distinct trigger dates in 12 months. Netty is quieter at 3 dates (fewer total deps, narrower attack surface). Under "rebuild on any critical/high," you're on a 2-3 week treadmill.

2. The NP+DI filter drops the noise floor by 67-93%. The degree varies — Django's recurring ORM SQLi keeps it at the lower end, Netty's single smuggling bug gives the highest reduction — but every stack gets meaningful breathing room.

3. Template engine injection showed up in 2 of 3 stacks. Thymeleaf SSTI in Spring, Handlebars code injection in Node.js. This is the most consistent direct-injection pattern across ecosystems — and exactly the vulnerability class with the strongest KEV track record.

4. The burst pattern is the real story. The filter doesn't produce a slower steady drip. It produces silence punctuated by genuine events. That's operationally different — it means when the signal fires, your team responds to it instead of tuning it out.

Django: The Honest Weak Case

Django is the weakest result, and that's worth understanding. Unlike Spring and Node.js (which had long silence windows), Django produced 4 NP+DI triggers in 12 months — roughly quarterly. All four were CWE-89 (SQL injection) in Django's ORM layer itself. The Django security team kept finding and patching variants of the same underlying issue (column alias handling, QuerySet connector kwargs).

This means the filter gives you less breathing room when the framework itself has a persistent vulnerability pattern in its parsing surface. But note: (a) every trigger was a genuine SQL injection — there were no false positives, and (b) 67% reduction is still operationally meaningful — you go from near-monthly fire drills to roughly bimonthly ones.

The counter-argument: "if I'm on Django, the filter only saves me 10 rebuilds a year." That's fair. But those 10 skipped rebuilds were DoS bugs, config issues, and non-NP library fixes that never reached KEV. The 4 rebuilds you did do were all real.

The Deserialization Question (Revisited)

Deserialization generated zero rebuild triggers across all three stacks.

Spring: jackson-databind (67 historical CWE-502 entries in OSV) produced zero C/H in 12 months. SnakeYAML: zero. The gadget-chain CVE factory has quieted.

Node.js: No deserialization-class vulnerabilities in any manifest package.

Django: pyyaml: zero. No deserialization events.

Deserialization is a CVE volume generator (historically), not an exploitation driver. In the KEV backtest, deserialization accounts for 3.8% of entries — and those are all product-level (Exchange, WebLogic, ColdFusion), not library-level. The real rebuild triggers were path traversal, template injection, and SQL injection.

April 2026 Spike

April was anomalous for Spring specifically. The Spring manifest saw 8 C/H events in April versus 14 in the prior 11 months combined. All 8 were in network-parser components (Tomcat, Thymeleaf). 6 of 8 were direct injection (including auth bypass via input manipulation).

The timing is suggestive: Anthropic announced Glasswing/Mythos on April 7. The Tomcat batch (7 CVEs) dropped April 9 — a coordinated release pattern consistent with accumulated findings. The Thymeleaf SSTI pair dropped April 15 — exactly the kind of pattern-matching vulnerability that AI-assisted code scanning excels at. No explicit Mythos attribution in any advisory, but the all-NP/zero-non-NP ratio in a manifest where parsers are only 33% of dependencies suggests focused scanning of parsing surfaces.

Node.js and Django did not show an April spike of the same magnitude.

Limitations & Caveats

OSV severity gaps. ~26% of OSV advisories lack parseable severity data. Some C/H events may be missed, though major ones (with GHSA or NVD backing) are captured.

CWE classification is imperfect. Some vulns classified as "not direct injection" might have injection characteristics under the right conditions. CWE-20 ("Improper Input Validation") is particularly ambiguous.

KEV lag. Zero KEV entries for these manifests in 12 months doesn't mean none of these vulns matter — it means CISA hasn't observed confirmed exploitation yet. The filter is predictive, not retrospective.

Transitive depth. Real apps pull far more transitive dependencies than modeled (Spring: 200+ JARs, Node.js: hundreds of npm packages). The full tree would push "all C/H" periodicity even shorter (more noise) while NP+DI events would grow proportionally less (more signal).

NP classification is judgment-call. We classified koa, fastify, handlebars, etc. as NP because they parse HTTP request/response data or render templates from potentially attacker-controlled input. Reasonable people could draw the line differently. The full CVE reference list shows every classification so you can audit and re-draw the line yourself.

Django's ORM SQLi pattern may not generalize. Django happened to have a persistent SQL injection issue during this 12-month window. A different 12-month window might show a different pattern. That said, the filter handled it correctly — it flagged every genuine SQLi.

The OS Layer: Vulnerability Chaining

The analysis above treats the application as the unit of patching. But production applications run on base images — a container OS with its own libraries, its own CVEs, and its own patch cadence. We took a real production container image (RHEL-based, ~450 executables) and ran the same 12-month NVD backtest to answer two questions: does the OS layer generate independent emergency rebuild triggers, and does it matter for blast radius when the app layer is breached?

The hacker discriminator was given the same Amazon Linux 2023 manifest with explicit context that this is the base image under the Spring stack. Two findings beyond the NP+DI = 0 result:

(1) After cleaning out CVEs that NVD keyword-matched into the wrong component, the 69 events shrink to ≈18 actual OS-component bugs. The other ~50 are CVEs in downstream apps that happen to share names with OS components — SQLite-using applications (FUXA, METIS, n8n, langchain, MyTube, SciTokens, PraisonAI, Cockpit, Kysely, SiYuan), kernel CVEs keyword-matched via systemd (drm/i915, wifi/brcmfmac, kernfs UAF, fs/writeback), Perl Compress::Raw::Zlib, undici WebSocket, snapd (which isn't even in Amazon Linux 2023), Incus container manager, and so on.

(2) Of the cleaned ≈18, the hacker S+A count is 0 strict (or 2-3 conditional-A if you count the kernel UAFs in the systemd cluster as post-compromise privesc paths into the host kernel). The B-tier candidates — libxml2 use-after-frees, libarchive heap bugs, pcre2 JIT OOB, OpenJ9 native stack overflow, zlib decompression chains — all gate on the Spring app having a native binding path to those libs (mostly it doesn't; Java apps use built-in javax.xml, java.util.regex, and JVM zlib). For this stack: 2-6 B-conditional, 6-8 C, 50+ D.

The structural finding is unchanged: the OS layer doesn't drive emergency rebuild cadence under either filter. What changes is the rationale for the regular container-refresh cadence below — it's about reducing real privesc surface (5-10 LPE-relevant bugs per year on components actually present in Amazon Linux 2023), not the inflated NVD count.

The Kill Chain: App Layer = IF, OS Layer = HOW BAD

But the OS layer isn't irrelevant — it determines blast radius after initial access.

If an attacker gains code execution through a Spring app vulnerability (say, Tomcat path traversal or Thymeleaf SSTI), the next question is: what can they do with it? The answer depends on what LOCAL privilege escalation paths exist in the container's OS layer. An unpatched container is an attacker's playground; a freshly patched one limits lateral movement.

This reframes patching into a two-layer model: the app layer determines whether you get breached (initial access), while the OS layer determines how bad the breach gets (blast radius through privesc).

Operational response model

Operational response depends on estate maturity — see Build Mechanics §5 for the full treatment.

Cat 1 — apps that absorb a security patch in hours via routine BAU rebuild.

Cat 2 — apps on a regular cadence (weekly to monthly). Discriminator-driven floor sweeps land cleanly. Most healthy estates live here.

Cat 3 — apps with a long, unscheduled, or governance-blocked rebuild path. Often larger than people admit. Every emergency becomes a project.

The discriminator's value depends on which category an app sits in. A Cat 1 estate barely needs the discriminator (everything gets absorbed in BAU). A Cat 3 estate needs it most (every false-positive emergency is weeks of project work). The hacker S+A bar of ~2.4 emergencies per year is the threshold below which a Cat 2 team can run on routine cadence and still cover real exploitation; the all-C/H bar of ~11 per year drives Cat 2 estates back into Cat 3 reactive mode.

The discriminator's second-order benefit: it makes the move from Cat 3 to Cat 2 possible. A team rebuilding every 2-3 weeks under all-C/H pressure can't credibly adopt a regular cadence — the cadence collides with the next emergency. The same team under hacker S+A pressure (2-3 emergencies per year on the production manifest) has months of slack between fires. That slack is where Cat 2 cadences get adopted, where get-latest build models get rolled out, where teams climb the maturity ladder. The reduction in rebuild churn is what unlocks the structural fixes; without it, the team is too busy firefighting to invest in not having to. See also Build Mechanics §9 (WAF as bridge).

The counter-argument worth considering: This model assumes LOCAL privesc CVEs are the primary chaining risk. But container breakout (escaping the container to the host) is arguably more impactful and usually requires different vulnerabilities (kernel bugs, container runtime flaws) not captured in this OS manifest analysis. The LOCAL CVEs here give escalation within the container — which matters for data access and lateral movement, but doesn't represent the full post-exploitation picture.

Additionally, the systemd CVEs dominate the LOCAL count (12 of 21), and some of these are kernel-matched CVEs that showed up in the systemd keyword search — NVD keyword noise inflates the count somewhat. The structural finding (OS layer doesn't drive emergency rebuilds, monthly refresh controls blast radius) is robust to this noise, but the specific numbers should be treated as approximate.

External Validation

The right question isn't "does the filter match what exploitation sources already know?" — it's "did we miss anything?"

A good prioritization filter doesn't need to be exactly right. It needs to be slightly conservative but never wrong — flag everything that matters, tolerate a few false positives, but never let a genuinely exploited CVE ride the slow cycle. Across 113 non-trigger CVEs, four ecosystems, and 12 months: zero misses. Nothing we deprioritized later showed up in KEV, got a Metasploit module, or accumulated meaningful exploitation evidence.

The filter may actually be a leading indicator — identifying CVEs that will accumulate exploitation evidence over the next 6-18 months, before the scanners and exploit developers get there. The Django SQLi that the filter flagged months ago is now showing up in ExploitDB and Nuclei templates. The Tomcat path traversal and Thymeleaf SSTI are exactly the vulnerability classes with the strongest historical KEV track records. The exploitation ecosystem just hasn't caught up yet.

We cross-referenced all 129 C/H CVEs against five independent data sources to test both directions — do triggers correlate with higher exploitation signals, and did any non-trigger accumulate evidence we should have caught?

EPSS is a retrospective tool being used for a real-time problem. Its scores are excellent for prioritizing a backlog of unpatched CVEs — but for the question "should I drop everything and rebuild right now?", it can't answer until the exploitation ecosystem has reacted. That takes days for famous vulns (Log4Shell), weeks for well-known libraries (Tomcat PUT), and months-to-never for quieter ones (XStream, ActiveMQ).

NP+DI is the opposite trade-off: it fires immediately because it reads the CVE's structure, not the world's reaction to it. The 3 misses are CWE metadata failures, not timing failures — and the DQ safety-net closes that gap on day zero too. The result is a filter that answers "rebuild now?" the moment the advisory drops, without waiting for Twitter to notice.

Conclusion

The filter works across ecosystems — and the OS layer confirms the model.

Under "rebuild on any C/H": the original three stacks each hit 14 trigger dates — one every 2-3 weeks. Netty is quieter at 3 dates. The treadmill never stops regardless of your tech stack.

Under "rebuild on NP + direct injection":

• Spring 5 events, 64% reduction. Triggers: Spring Security auth bypass (×2), Tomcat CLIENT_CERT auth failure, Tomcat path traversal, Thymeleaf SSTI.

• Node.js 2 events, 10 months of silence before first hit, 86% reduction. Triggers: Koa host injection, Handlebars code injection.

• Django 6 events, 67% reduction. Triggers: Django ORM SQL injection (×5), PyJWT crit header validation bypass.

• Netty 1 event, 360 days of silence, 67% reduction. Trigger: Netty HTTP request smuggling (CWE-444). Without the CWE-444 addition, this stack would have had zero triggers.

The OS container layer produced zero NP+DI events in 12 months — confirming that the app layer drives emergency rebuild cadence, not the base image. But the OS layer isn't free: 21 LOCAL privesc CVEs accumulated over the year. Monthly container refresh keeps that exposure at 5 instead of 21, reducing blast radius by 76% without requiring emergency response.

Everything else (Multer DoS, urllib3 decompression bombs, Pillow buffer overflows, Tomcat info leaks, Fastify validation bypasses) rides the team's normal update process and gets swept up automatically — exactly which cadence depends on the team's estate-maturity category. Zero of the 34 skipped app-layer events appeared in KEV, so the timing of the catch isn't urgent. Over the 7-year real manifest, NP+DI and EPSS ≥ 0.50 produce similar patch-event counts (~19 vs ~21) but catch different things — NP+DI fires on day zero before exploitation signal builds; EPSS catches CWE-misclassified misses. The practical model may be NP+DI as the structural baseline with EPSS as a second-look safety net.

The result is a layered response: emergency response for app-layer NP+DI / hacker S+A triggers (2-6 times/year), regular container-refresh cadence for OS base images (blast radius control), and the team's normal update process for everything else — see Build Mechanics §5 for how this maps onto Cat 1 / Cat 2 / Cat 3 estate maturity, and §9 for how WAFs bridge gaps when an estate can't move up the categories cheaply. The filter preserves the "seatbelt not airbag" principle — you still patch everything in due course, but you stop treating every C/H as a fire drill. And by getting emergency-rebuild churn under control, the team gets the bandwidth to climb from Cat 3 to Cat 2 on the maturity ladder — the discriminator's second-order win.
