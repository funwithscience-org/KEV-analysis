Where Active Exploits Actually Land

An empirical analysis of 1,568 CISA Known Exploited Vulnerabilities against 23,501 critical/high NVD CVEs. The goal: move beyond CVSS scores to data-driven vulnerability management prioritization.

Open Interactive Dashboard →

1. The Hypothesis

The starting premise was simple: most actively exploited CVEs cluster at the platform and infrastructure layer — operating systems, VPN appliances, JVM/web servers, and SSL/TLS stacks. Individual open-source library criticals, despite generating scanner noise, are rarely exploited at scale. Log4j was the notable exception.

As the analysis progressed, a sharper formulation emerged: the real predictor isn't "platform vs library" — it's "does this component parse untrusted network input?" Platform components happen to do this by default (that's what makes them platforms), while most libraries sit behind layers that handle parsing for them.

This reframing matters because it changes how you triage. A library that directly parses HTTP (like Struts or Spring MVC) inherits the risk profile of a platform component — and the data confirms this.

2. Methodology

The analysis joins two datasets:

CISA KEV catalog — 1,568 vulnerabilities with confirmed active exploitation. This is the "ground truth" for what attackers actually target. It's curated by CISA for US federal agency compliance, which introduces selection bias toward enterprise/government targets.

NVD (National Vulnerability Database) — 23,501 CVEs rated Critical or High (CVSS 7.0+) published 2021–2026. This is the denominator: the full universe of theoretically exploitable vulnerabilities.

Each CVE was categorized by stack layer (OS, VPN/network appliance, JVM/runtime, web server, browser, email/collaboration server, library/framework, CMS/webapp, productivity/desktop, database, firmware/hardware, virtualization/container, IoT/ICS, SSL/TLS/crypto) using CPE product strings and description pattern matching. CWE IDs were grouped into families (memory corruption, injection, auth, path traversal, deserialization, etc.).

The exploitation rate for each category = (KEV count in category) / (NVD count in category). This tells us: given a critical/high CVE in category X, what's the probability it ends up actively exploited?

Libraries: The Survivorship Illusion

The initial categorization found only 80 library CVEs in NVD and 21 in KEV, producing a misleading 26.3% exploitation rate. After expanding the library definition with comprehensive product matching (~200 known library product names), we identified 345 library CVEs in NVD with 8 in KEV.

The corrected library exploitation rate is 2.3–4.4% depending on severity level — still 2–4× the overall baseline of ~1.1%. Libraries aren't "safe"; they're a small population. The ones that do get exploited tend to be the same handful: Apache Struts (6 KEV entries), Spring ecosystem (4), ImageMagick (3), Log4j (2), Ruby on Rails (3).

Nuance: Per-CVE, libraries that parse HTTP are exploited at rates comparable to platform components. The low absolute count (21 KEV entries) creates a false sense of safety. A single library like Struts accumulates 6 KEV entries over time — that's a recurring, not a one-off, threat.

4. HTTP-Parsing as Exploitation Predictor

The sharpest signal in this data isn't CVSS severity — it's whether the vulnerable component directly parses untrusted network input (primarily HTTP requests, but also DNS, SMTP, SSH, etc.).

We classified each NVD CVE as "HTTP-parsing" (product known to handle untrusted HTTP/network input: web servers, app servers, VPN appliances, email servers, browsers, CMS platforms, Struts, Spring MVC, etc.) or "non-HTTP" (everything else). This makes intuitive sense. CVSS measures theoretical impact and exploitability in a vacuum. But attackers operate in the real world, where the first question is: "can I reach it from the internet?" Products that parse HTTP are, by definition, reachable. Products that don't require additional access — which most attackers don't have yet.

The Reverse Proxy Myth

A common pushback: "Our apps sit behind Apache/nginx reverse proxy, so library vulnerabilities are mitigated." This is partially true for network-layer attacks but largely false for application-layer exploits.

Spring4Shell (CVE-2022-22965) exploits a class-loading manipulation via HTTP request parameters. mod_proxy forwards those parameters untouched. Log4Shell (CVE-2021-44228) exploits JNDI injection via HTTP headers — the User-Agent or X-Forwarded-For header containing ${jndi:ldap://...} passes through any reverse proxy. The proxy doesn't parse or inspect the application-layer semantics of what it forwards.

A Web Application Firewall (WAF) with signatures can help — but only for known exploits. Zero-days bypass WAFs by definition. And signature updates lag behind exploit publication by hours to days, which — as we'll see in the time-to-exploit section — is now the entire exploitation window.

What the Reverse Proxy Can Do

The above isn't an argument against reverse proxies — it's an argument against relying on them as your library vulnerability mitigation. A properly configured reverse proxy still provides real defense-in-depth, just not the kind most people assume. Here's what actually works:

Path constraint / URI allowlisting. The single most effective reverse proxy control. If your app only serves /api/v1/* and /static/*, configure the proxy to reject everything else at the edge. Spring4Shell's exploitation relied on accessing /... class-loader paths — a path allowlist blocks this even without knowing the CVE exists. This is the closest thing to a "free" mitigation because it constrains the attack surface to only the routes your app explicitly exposes. The key is to default-deny and allowlist, not default-allow and blocklist.

Request size and depth limits. Cap request body size, header count, header size, query string length, and URL depth. Many exploitation payloads (especially deserialization attacks) require oversized or deeply nested request structures. A 16KB body limit and 8KB header limit won't stop Log4Shell's short JNDI string, but they will stop many deserialization and file upload exploits. The protection is probabilistic, not absolute — but it raises the bar.

Header stripping and normalization. Remove or rewrite headers your application doesn't need. If your backend doesn't use X-Forwarded-For for anything, strip it at the proxy — this would have neutered one of Log4Shell's common injection vectors. Strip Transfer-Encoding: chunked if your app doesn't need it (HTTP request smuggling mitigation). Normalize Content-Type to prevent deserialization attacks that rely on unexpected media types.

TLS termination. Terminates the encrypted channel at the proxy and re-encrypts to the backend on a controlled channel. This ensures the proxy can actually inspect traffic. Without TLS termination, the proxy is just a TCP forwarder and can't apply any of the above controls.

Response header hardening. Remove server version banners (Server: Apache/2.4.52), framework identifiers (X-Powered-By: Express), and stack traces from error responses. This doesn't prevent exploitation of known CVEs, but it forces attackers to probe blindly rather than selecting exploits based on your advertised versions.

The practical takeaway: A reverse proxy configured with path allowlisting + header stripping + request size limits provides meaningful defense-in-depth against library exploits — not by understanding the vulnerability, but by constraining what the attacker can send to a surface small enough that many exploit payloads simply can't reach the vulnerable code path. The key word is "constrain," not "prevent." It's a seatbelt, not an airbag.

5. Why Most Criticals Don't Get Exploited

If there are 23,501 critical/high CVEs and only 1,568 in KEV (~6.7% naively), what makes the other 93% not worth exploiting? Six patterns explain the vast majority:

Firmware and hardware (3,739 NVD, 5 KEV, 0.13%). Chipset BIOS, printer firmware, UEFI modules — these get high CVSS scores because the theoretical impact is complete system compromise, but exploitation requires physical access, specific device models, and often a pre-existing foothold. The airgap and update friction make them poor targets for mass exploitation.

Linux kernel local privilege escalation. Many kernel CVEs score critical but require local access. Attackers prioritize remote-first vectors for initial access; local privesc is a second-stage concern, often solved with commodity tools rather than specific CVEs.

Consumer IoT and embedded devices. Smart home devices, IP cameras, consumer-grade routers — high CVSS, low strategic value. Nation-state and sophisticated criminal actors focus on enterprise infrastructure.

Non-Western ecosystem products. Products popular in specific regional markets (Huawei, Tenda, certain Chinese CMS platforms) accumulate NVD criticals but have limited presence in the Western enterprise networks that KEV primarily tracks.

Memory corruption complexity. Many heap overflow and use-after-free CVEs score 9.8 but require precise heap grooming, specific binary versions, and ASLR/DEP bypasses. The skill barrier filters out most attackers.

One-CVE-wonder products. Niche products with a single critical CVE and minimal deployment. The ROI of developing an exploit for a product used by 200 organizations doesn't justify the effort.

Implication: These patterns are predictable. If a CVE falls into one of these categories, the historical exploitation probability drops to near-zero. This isn't a reason to never patch — it's a reason to deprioritize relative to the categories that actually get exploited.

The window between CVE publication and active exploitation has collapsed dramatically.

The new reality: Median time-to-exploit dropped from 251 days (2021) to 11 days (2023). The P25 — meaning the fastest quartile of exploited CVEs — hit 3 days. One quarter of actively exploited vulnerabilities are weaponized within 72 hours of disclosure. A 30-day patch cycle now leaves you exposed for the entire exploitation window.

The 2024 data (median 34 days) appears to reverse the trend, but this is a right-censoring artifact: many 2024 CVEs haven't been added to KEV yet because exploitation hasn't been confirmed or reported. The true 2024 median will likely settle lower as more entries are added.

The compression is driven by several factors: the commoditization of exploit development (n-day exploit kits), faster reverse engineering from patches, broader information sharing in both offensive and defensive communities, and the industrialization of ransomware operations.

7. Finance Sector KEV Blind Spots

KEV is curated for US federal agency compliance. This creates structural blind spots for financial services, which face a different threat landscape:

Active Directory / Kerberos / LDAP (~16 KEV entries). AD is the backbone of every enterprise network and the primary post-compromise target. Despite being the most attacked identity infrastructure in finance, it has minimal KEV representation because AD attacks often use features (Kerberoasting, Golden Tickets) rather than CVEs.

Middleware and message queues (~3 KEV entries). Apache Kafka, IBM MQ, RabbitMQ, TIBCO — these form the backbone of financial messaging and trading systems. Nearly invisible in KEV, yet a compromised message bus could manipulate transactions at scale.

SWIFT and ISO 20022 (0 KEV entries). The Bangladesh Bank heist (2016) demonstrated that financial messaging systems are targets for nation-state actors. SWIFT-specific vulnerabilities wouldn't appear in KEV because they're specialized, not widely deployed, and exploited by actors who don't leave public indicators.

Mobile zero-click exploits (~13 KEV entries via Samsung/Apple). Executive-level targeting via NSO Group-style attacks. Underreported because exploitation is designed to be undetectable, and the targets (C-suite, regulators) don't typically trigger CISA reporting.

Living-off-the-land techniques. PowerShell, WMI, and other built-in OS tools used post-compromise don't map to CVEs at all. They're the most common method for lateral movement in financial networks, yet entirely absent from vulnerability management frameworks.

For finance teams: KEV is necessary but not sufficient. Supplement with industry-specific threat intelligence (FS-ISAC), red team findings, and monitoring for technique-based attacks (MITRE ATT&CK) that don't have CVE identifiers.

This reinforces a pattern: attackers favor high-impact, easy-to-trigger vulnerabilities over technically sophisticated ones. Of the 1568 KEV entries, 313 (20%) are linked to known ransomware campaigns. Ransomware operators concentrate on the same platform/infrastructure targets: OS, VPN appliances, and email servers — aligning with the broader exploitation pattern.

For the Urgent tier, the specific products to monitor intensively include: Apache Struts, Spring Framework/Boot/Cloud, Log4j, ImageMagick, Ruby on Rails, and any application framework your organization uses that directly handles HTTP request parsing. These are the known library exceptions that behave like platform components from an exploitation perspective.

10. Mythos Detector: Will AI Break the CVE Firehose?

On April 7, 2026, Anthropic publicly announced Project Glasswing — an initiative providing its unreleased Mythos Preview model to major technology companies so they can autonomously scan their own codebases for vulnerabilities. Participants had access to Mythos Preview before the public announcement and were already finding and fixing bugs. Anthropic provides the tool; the vendors do the scanning, discovery, patching, and disclosure through their own normal processes. The question for vulnerability management teams is immediate: will this meaningfully increase the monthly volume of CVEs or exploits you need to deal with?

What Mythos Has Found So Far

According to VulnCheck analysis of the CVE database, approximately 40 CVEs can be tentatively linked to Anthropic or Anthropic-affiliated researchers. Three now carry explicit Claude/Mythos credit lines:

  • CVE-2026-4747 — FreeBSD NFS RCE (17-year-old bug). Confirmed autonomous Glasswing find.
  • CVE-2026-5194 — wolfSSL certificate-validation bypass (~5B affected devices). Press-confirmed as Mythos Preview–assisted discovery, credited to Nicholas Carlini at Anthropic.
  • CVE-2026-5588 — Bouncy Castle cryptographic library. Credited to Carlini + Claude.

The pattern has moved beyond a single data point: three Claude-credited disclosures in two weeks, spanning two TLS/crypto libraries and one OS kernel, suggests a real — if still early — signal rather than noise.

Beyond the explicitly credited CVEs, we're also tracking probable Glasswing participant self-scan findings — CVEs from named Glasswing participants that meet all of the following criteria: the vendor is a participant, the bug is HTTP-parsing-adjacent, there was no attacker exploitation before disclosure, no third-party researcher credit in the advisory, and the bug pattern is consistent with automated AI code scanning (input validation, logic flaws, certificate handling — the kinds of things AI excels at finding systematically across large codebases).

The strongest signal so far is a Cisco cluster — four critical vulnerabilities disclosed in a single batch on April 15, 2026. Three ISE RCEs via crafted HTTP requests hitting the same input-validation bug class across different code paths, plus a Webex SSO certificate-validation bypass. Cisco is a named Glasswing participant. The cluster pattern — multiple related bugs in the same component found simultaneously — is a hallmark of systematic automated scanning rather than ad hoc human auditing.

Projection: Will It Move the Needle?

Monthly CVE Trend with Projected Glasswing Impact Scenarios

Bottom line: Even the conservative scenario (+500 CVEs/month from Glasswing) would push monthly volume past 6,000 sustainably. The aggressive scenario (if Mythos finds vulnerabilities at the rate the Firefox benchmarks suggest) could add 2,000–5,000 CVEs/month, pushing total volume to 8,000–11,000 — a level that would break most organizations' patch management processes.

The critical question isn't just volume but severity distribution. If Mythos disproportionately finds critical/high vulnerabilities in platform-layer software (as the Firefox and kernel benchmarks suggest), then the "urgent" tier of our patch SLA framework would grow dramatically. Rich Mogull's warning that we could face "multiple Log4j-level events every month" is not hyperbole — it's an extrapolation from the benchmark data.

However, there's a counterargument worth considering: Glasswing is explicitly a defensive initiative. These CVEs will be discovered and patched by the vendors themselves before disclosure. The net effect could actually be positive for defenders: a one-time surge in patches for long-latent bugs, followed by a cleaner codebase. The danger scenario is if Mythos-class capability proliferates to offensive actors before the defensive cleanup is complete — and the February 2026 report of Claude Code being used in attack chains across ~30 global targets suggests that timeline is already compressed.

Full Glasswing results are expected around July 2026, but the volume spike is likely to arrive earlier. Participants are racing to fix preview-era findings before production Mythos ships and the same bugs become discoverable by anyone with access. Expect the conservative scenario to begin materializing in May 2026, with the aggressive scenario's inflection as early as May–June rather than the original July estimate.

11. Exploit Watch List — HTTP-Parsing Attack Surface

If the HTTP-parsing thesis holds — that components directly handling HTTP request/response parsing are exploited at 3-6x the rate of non-HTTP components — then we should be able to predict which newly disclosed CVEs are most likely to end up in KEV. This section tracks those predictions in real time, split by server-side and client-side attack vectors.

The focus is deliberately narrow: web servers, template engines, TLS terminators, reverse proxy components, API frameworks, and browser-delivered payloads that traverse the HTTP parsing stack. Network device firmware, kernel privilege escalation, and other non-HTTP vectors are tracked elsewhere but excluded from this watch list.

Selection Criteria

A CVE is added to the watch list when it meets at least three of these five indicators:

1. HTTP-parsing adjacency. The vulnerable component directly processes, renders, or terminates HTTP traffic — request parsing, response rendering, TLS handshake, header processing, template execution, or content delivery.

2. Low or no authentication requirement. Unauthenticated or pre-auth exploitation paths. Our data shows that CVEs requiring authentication have significantly lower KEV representation.

3. High exploitability signal. CVSS attack complexity Low, public PoC available, or pattern-match to a previously exploited bug class (e.g., SSTI, deserialization, path traversal via archive extraction).

4. Broad deployment footprint. The affected product is deployed at scale — Spring ecosystem, NGINX/Apache modules, embedded TLS libraries, major CMS platforms.

5. CWE family match. The vulnerability falls into a CWE family with historically high KEV representation: injection (CWE-94/78/89), memory corruption (CWE-787/416/122), authentication bypass (CWE-287/306), or deserialization (CWE-502).

Thesis Challenge: Non-HTTP Library Exploits

Good analysis requires actively searching for disconfirming evidence. The HTTP-parsing thesis predicts that library-layer and framework-layer CVEs reaching confirmed exploitation (KEV) will overwhelmingly involve HTTP-adjacent attack surface — parsing untrusted network input, handling HTTP requests, terminating TLS, rendering templates, or processing browser-delivered payloads. If the thesis is correct, we should rarely see library/framework CVEs exploited in the wild via purely non-HTTP vectors.

This section tracks the counter-examples: library or framework CVEs that reach confirmed exploitation through attack vectors that have nothing to do with HTTP parsing. We exclude OS-level and kernel CVEs (which have near-100% exploitation rates regardless of vector) and commercial product CVEs (which are outside the library/framework layer). We're specifically looking for things like: a math library exploited via local file processing, a serialization framework exploited via non-HTTP protocols, or a crypto library attacked through a non-TLS channel.

Every entry here weakens the thesis. An empty list strengthens it — but may also reflect detection bias, since non-HTTP exploitation in libraries is harder to observe than internet-facing attacks.

What would falsify the thesis? A sustained pattern of library/framework CVEs reaching KEV via non-HTTP vectors — e.g., a message queue library exploited over AMQP, a data-processing library exploited via file ingestion, or a serialization framework exploited over a custom binary protocol. A single counter-example is a data point; a cluster would be a genuine challenge.

12. Caveats & Methodology Limitations

KEV selection bias. CISA KEV prioritizes US federal interest and publicly confirmed exploitation. Private-sector exploitation, especially by sophisticated actors, is underrepresented. A zero-day in a niche financial library might never reach KEV even if actively exploited against banks.

NVD denominator quality. CPE mapping is imperfect — products like "Microsoft" appear without precise product boundaries. Approximately 17,000 CVEs fell into "other" due to sparse metadata. Severity and CVSS fields were sometimes missing and required re-extraction from raw JSON.

Library detection limits. We classified libraries by product name pattern matching (~200 known products plus vendor sets). Obscure npm/PyPI/RubyGems packages may be undercounted. However, the high-impact libraries that actually get exploited (Struts, Spring, Log4j, ImageMagick, Rails) were all captured.

Time-to-exploit right-censoring. Recent years (2024–2026) show artificially higher medians because many CVEs haven't been added to KEV yet. The true time-to-exploit for recent years will settle lower over time.

HTTP-parsing classification. The binary HTTP/non-HTTP split is a simplification. Some "non-HTTP" products (like databases exposed to the internet) can be directly attacked, and some "HTTP" products may sit behind additional security layers. The signal is directional, not absolute.

Despite these limitations, the core findings are robust: platform/infrastructure dominates active exploitation, HTTP-parsing is a stronger signal than CVSS, and time-to-exploit has compressed to days. These patterns hold across multiple cuts of the data.
