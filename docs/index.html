Threat-Centric Prioritization — A New Way of Managing Vulnerabilities

For third-party products — Cisco, Microsoft, Palo Alto — patching is a solved problem. Vendors ship fixes on a known schedule, you apply them, done. The hard problem is your own application code, sitting on top of 50+ open-source libraries with no coordinated patch schedule. Every week brings a new critical. Each one means a rebuild, a test cycle, a deploy. You can't do them all. So which ones actually matter?

The Problem and the Proposal

The problem. Vulnerability volume is exploding. Even restricting to "criticals and highs," the count runs into the thousands per year — but only ~4% of those ever actually turn into exploits. For third-party packages, updating binaries and rebooting is relatively straightforward. For internally built systems that depend on dozens of open-source libraries, managing the rebuild and release process is substantially more complex.

The proposal. We believe we have produced a model — call it threat-centric prioritization — that is substantially better at projecting which vulnerabilities will end up being exploited. The core idea: judge a CVE by what an attacker actually does with it, not by what its severity score says. The model has two complementary but independent operationalizations, each developed against the same data:

  • The structure test (NP+DI). A mechanical filter on the characteristics of the library and the CVE: does the package parse untrusted input, and is the bug a direct-injection class? Easy to apply, hard to argue with, doesn't depend on judgment.
  • The attacker test (hacker discriminator). The same tools and capabilities attackers use — notably modern AI applied as an offensive operator's grading rubric — to evaluate vulnerabilities by exploitability and value. Catches the same population the structure test does, with informative disagreements at the boundary.

This substantially reduced list of high-risk events becomes the mandatory rebuild trigger. It is supplemented by additional security controls and process improvements:

  • "Get newest" build methodologies — routine pipelines that absorb non-trigger patches automatically.
  • Better application metadata capture — knowing which packages are deployed, in which configuration, behind which controls.
  • Targeted WAF deployment — much more viable now that the attack vector is focused.
  • Continued monitoring of vulnerability intelligence — KEV, EPSS, ExploitDB, Nuclei, Metasploit and successor feeds, as a safety net for anything the model misclassifies and as the early-warning system for novel attack patterns.

The Problem

In plain English: Patching third-party products is well-understood, even if it's tedious. Patching applications you build yourself, on top of dozens of open-source libraries, is the actually-hard problem.

For third-party vendor products — your Cisco switches, your Windows servers, your Palo Alto firewalls — patch management is a solved-if-tedious problem. Those vendors ship patches on known schedules, your ops team knows how to turn the crank, and the process is well-understood. There is a Patch Tuesday and there is a runbook.

Your own application code is a different story. A modern application might depend on 50+ open-source libraries, each maintained by different people, each shipping CVEs on their own schedule. There is no Patch Tuesday for open source. When Spring, or Tomcat, or Express, or Django ships a critical, you have to decide: do I drop everything, rebuild, retest, and redeploy right now? Or can this ride the next scheduled release?

The default answer — "rebuild on any critical/high in my dependency tree" — sounds responsible, but it doesn't survive contact with portfolio scale. Across three different tech stacks the rule triggers 14 emergency rebuilds per year. For one application in isolation that may not sound crazy — many teams are doing builds more often than that anyway. The chaos comes from running the same rule across hundreds of applications simultaneously, where applications break into roughly three operational profiles:

  • Active-development apps. Continuously evolving codebases with mature pipelines. They can absorb 14 unscheduled rebuilds a year with varying levels of discomfort — the most common pushback is the cost of integrating substantial business changes and tech changes at the same time. (Not necessarily a valid pushback, but a common one.)
  • Infrequent-development apps. Modest change volume, often driven by regulatory or external requirements. The team does ship builds, but at a much lower cadence than the active ones, and an unscheduled emergency rebuild cuts more sharply against their normal flow.
  • Stable / stale apps and components. Code that hasn't shipped a meaningful change in months or years. Every emergency rebuild is effectively a project — the pipeline may need reviving before it can be used, dependencies may have drifted out of step with the rest of the platform, and there's no recent baseline to validate against.

Every organization has a different mix of these three based on industry, history, and funding. And because you usually can't tell from the outside which apps fall into which profile without good metadata (the application-metadata problem we flagged in the proposal above), the safe default becomes treating the whole portfolio as if it's all stable-and-emergency-grade every time a critical lands.

The cost is self-reinforcing. The firefighting churn that comes from forcing the slowest-profile response onto the whole portfolio is exactly what crowds out the structural fixes — get-newest builds, regular cadence, application metadata, targeted WAF — that would move apps from the third profile toward the first. The discriminator's value isn't "fewer rebuilds for one team"; it's fewer apps swept into the slowest path at portfolio scale, which buys back the bandwidth to actually fix the underlying maturity problem.

So the operational question is: which of those 14 triggers actually matter? Is there a signal in the data — something more useful than the CVSS score — that separates the CVEs attackers actually exploit from the ones that just make noise? Two independent answers come out of the analysis, and they largely agree: network parser + direct injection (a CWE-based mechanical filter) and the hacker discriminator (an attacker's grading rubric applied with modern AI tooling). The rest of this page is how we got there and how well it holds up against ground truth.

The First Clue: Where Exploits Actually Land

In plain English: We took every vulnerability that attackers actually exploited (1,583 in the KEV catalog, 891 with CVEs published 2021–2026) and compared it to every serious vulnerability reported in the same window (23,501). Then we asked: what makes the exploited ones different?

The analysis joins two datasets:

CISA KEV catalog -- 1,578 vulnerabilities with confirmed active exploitation (886 with CVEs published 2021–2026; exploitation rates use these windowed entries to match the NVD denominator). This is the "ground truth" for what attackers actually target. It's curated by CISA for US federal agency compliance, which introduces selection bias toward enterprise/government network-facing targets (see Caveats).

NVD -- 23,501 CVEs rated Critical or High (CVSS 7.0+) published 2021-2026. This is the denominator for cross-layer comparisons — it's the only source that consistently covers OS, VPN, browser, and library categories on the same scale.

Important caveat for teams that build their own code: NVD massively under-represents open-source library vulnerabilities. Most small libraries never register a CPE, so their CVEs are invisible to NVD's categorization. Google's OSV database — which aggregates NVD, GitHub Security Advisories, and ecosystem-specific sources — tracks 770 critical/high vulnerabilities across just 177 popular packages, more than double NVD's 345 for all libraries. The true library vulnerability population is orders of magnitude larger than NVD suggests. This means the noise your application team faces is significantly worse than the NVD-based numbers in this analysis imply. We use NVD as the primary denominator for the cross-layer comparison (§2) because it's consistent across categories, and switch to OSV for the library-specific and periodicity analysis (§2–8) where accuracy matters more than cross-category consistency.

Each CVE was categorized by stack layer (OS, VPN/network appliance, JVM/runtime, web server, browser, email/collaboration server, library/framework, CMS/webapp, productivity/desktop, database, firmware/hardware, virtualization/container, IoT/ICS, SSL/TLS/crypto) using CPE product strings and description pattern matching. The exploitation rate for each category = (KEV count) / (NVD count).

2c. Libraries: The Denominator Makes the Difference

NVD tracks roughly 345 critical/high library CVEs for our period -- and at that denominator, the 21 library KEV entries imply a 2.3–6.1% exploitation rate. Sounds significant. But NVD massively undercounts libraries. OSV tracks 770 critical/high vulnerabilities across just 177 popular packages alone -- more than double NVD's total. Across the five major ecosystems (npm, PyPI, Maven, Go, RubyGems), OSV lists roughly 247,000 advisories total, with an estimated 93,000+ at critical/high severity. Against that real denominator, the exploitation rate drops to well below 0.1%. Your scanner flags thousands of library CVEs per year; virtually none of them will ever be exploited in the wild.

Of those 177 popular packages, 81 (46%) have zero critical/high vulnerabilities -- and 70 of those 81 are non-network-parser libraries. The median network-parser library has 3 critical/high advisories; the median non-network-parser has zero. The top 10 most-vulnerable packages hold 62% of all critical/high advisories, and 6 of 10 are network parsers.

The math that matters: At NVD's denominator (345 C/H library CVEs), 4 windowed KEV entries looks like a ~1% exploitation rate. At OSV's denominator (770 C/H across 177 packages, ~93,000+ ecosystem-wide), the same 4 KEV entries are a rounding error. The library exploitation rate isn't "low" — it's insignificant. The network-parser signal is what separates the handful that matter from the thousands that don't.

So with libraries the problem isn't finding the 4%. It's finding the <0.1% that actually get exploited and not getting buried in the volume. This is the opposite of the third-party-product problem, where the base rate is high enough that "patch everything from this vendor" is a defensible default. With open-source libraries, "patch everything" isn't responsible — it's the response that guarantees you can't tell signal from noise.

Threat-Centric Prioritization: Two Independent Models

In plain English: The hero isn't a single filter — it's a model. Judge a vulnerability by what an attacker actually does with it, not by what its CVSS says. We've built two competitive ways to operationalize that model. They run independently against the same data, mostly agree, and where they disagree is also informative.

At this point the data tells us two things: network-parsing components get exploited at 3–6× the rate of everything else (§2), and the exploitation window is measured in days, not months (§6). But "rebuild on any critical/high in a network parser" is still too noisy to be useful. That rule fires 14 times in 12 months across every stack we tested — about every 2–3 weeks. You'd be churning almost as badly as before.

The reason it's too broad: most network-parser CVEs aren't the kind of bug attackers actually exploit for initial access. A denial-of-service in Tomcat is a real bug, but it doesn't give an attacker code execution. An authentication-bypass in Spring Security is concerning, but it requires specific configuration to be reachable. A Fastify validation-bypass is a defense-in-depth issue, not a front-door entry point. What actually reaches KEV — what attackers build exploits for and use in the field — is overwhelmingly the direct primitive: path traversal, template injection, SQL injection, command injection, request smuggling, deserialization-at-default. Bugs where untrusted input becomes executable code or crosses a security boundary without mediation.

That insight is the model: threat-centric prioritization — pick the events worth interrupting cadence for by what the threat actually looks like, not by what severity score the CWE was assigned. The model has two operational forms in the rest of this page, each developed independently against the same data:

  • The structure test — NP+DI (§3a–§3c) — a mechanical filter on the data itself: does the package parse untrusted input and is the bug a direct-injection class? Both yes → fire. Easy to apply, hard to argue with, doesn't depend on judgment.
  • The attacker test — hacker discriminator (§3d) — an offensive operator's grading rubric using the same modern AI tools attackers do: default config × network edge × primitive direct, with auth-missing as a co-tag. Catches the same population the structure test does, with small but informative disagreements at the boundary.

Periodicity backtest. To validate threat-centric prioritization, we ran both operationalizations of the model — the structure test (NP+DI) and the attacker test (hacker discriminator) — independently against several representative application manifests plus an OS container layer, and measured how often each would trigger an emergency rebuild. We then cross-validated against five independent exploitation sources: EPSS, CISA KEV, ExploitDB, Nuclei, and Metasploit. Full methodology, per-manifest detail, and per-CVE classifications are in the periodicity analysis and CVE reference.

3a. The structure test — NP+DI

Two criteria, both required:

1. Network Parser (NP). The vulnerable component directly parses untrusted network input — HTTP requests, TLS handshakes, DNS responses, SMTP headers, template expressions, authentication tokens. This includes web frameworks (Spring, Express, Django), app servers (Tomcat, Gunicorn), template engines (Thymeleaf, Handlebars, Jinja2), HTTP client libraries, WebSocket handlers, GraphQL servers, and TLS libraries. It does not include ORMs that only receive pre-parsed input (Sequelize, Hibernate), utility libraries (lodash, guava), build tools, or local-only processing libraries (Pillow, numpy).

2. Direct Injection (DI) CWE. The vulnerability carries a CWE from a specific set — the bug classes where attacker input becomes executable code or traverses a security boundary. Not every network-parser bug qualifies. DoS, info leak, and configuration issues are real problems, but they're not the bugs that give attackers a foothold.

Notably absent: CWE-502 (Deserialization). The reasoning needs care, because at the library level deserialization absolutely can be an exploitation driver (XStream pre-1.4.18, SnakeYAML constructor deser, the Tomcat partial-PUT chain — see §4f). What the data shows is that raw CWE-502 volume hugely overstates how often deserialization is actually reachable. Most CWE-502 disclosures are gadget-chain-dependent: they require very specific classes to be on the classpath, very specific application code to call the unsafe API on attacker-controlled bytes, and a deployment shape that exposes the path. Jackson-databind alone accumulated 67 historical CWE-502 entries; a rounding-error fraction reached KEV. So CWE-502 is excluded from the mechanical DI filter — adding it would drown the real signal in noise. Catching the exploited subset of deserialization needs a different cut of the same data, which we develop next. Also absent from the DI set: CWE-400 (Resource Exhaustion / DoS) — denial of service doesn't produce code execution.

3c. Why the Filter is Testable and Falsifiable

The NP+DI filter makes a specific, falsifiable prediction: critical/high CVEs that fail the filter (non-NP, or NP but non-DI CWE) will not appear in CISA KEV within 12 months of disclosure. Any counter-example weakens the filter. A cluster of counter-examples kills it.

This is operationally important because it means you can measure the filter's performance against your own dependency stack. Run it for 6 months. Track what it flags and what it skips. If something it skipped gets exploited, the filter needs adjustment. If nothing it skipped gets exploited, you've validated it for your environment. The CVE reference list shows every classification so you can audit and re-draw the line yourself.

3d. The attacker test — the hacker discriminator

In plain English: NP+DI is a CWE-and-package rule — mechanical, easy to apply, hard to argue with. But you can also frame the same triage decision the way an attacker would: "would I actually invest in writing an exploit for this?" That gives a second, independent filter to cross-check NP+DI against.

An offensive operator looking at a CVE feed asks three questions before committing time to an exploit campaign:

1. Default config? Does this bug fire on a vanilla install of the affected software, or does it require some non-default option to be set? A bug that needs readonly=false on a Tomcat that ships readonly=true by default is a much narrower target than a bug that fires on the as-shipped configuration.

2. Network edge? Is the bug reachable from the internet via the path the application normally exposes, or is it behind some internal-only surface (intra-cluster, mTLS-only, dev-tool-on-laptop)? An HTTP-edge bug is universal; an AJP-on-port-8009 bug is reachable only when port 8009 is exposed.

3. Primitive direct? Does the bug give code execution / data exfiltration / auth bypass directly, or does it require chaining into a second bug or attacker-side gadget that may or may not exist? Direct primitives are an order of magnitude more valuable than chained ones because the chain can break.

Apply those three filters to a CVE feed and grade tier:

  • S — Hits all three: unconditional one-shot at default config, internet-edge, direct primitive. Log4Shell. Ghostcat. SnakeYAML. Famous-bug shape.
  • A — Hits two of three with the third near-default. Spring4Shell, Text4Shell, partial-PUT chain — strong primitive but with a precondition the attacker can usually find.
  • B — Strong primitive but real precondition gating reach (Tomcat path traversal — non-default config; Tomcat HTTP PUT — needs readonly=false).
  • C — Defense-in-depth weakening, MITM-required, or chain-into-only.
  • D — DoS, info-leak, weak crypto config. Real bugs, but not exploitation primitives.

The hacker S+A bar is "events I'd actually put on my campaign list." It produces a different cut of the same CVE feed than NP+DI: more selective in some ways (ignores SQLi clusters with cluster-mean preconditions), more aggressive in others (catches Spring Boot actuator exposure that CWE-862-tagged events would slip past NP+DI on). On the 7-year production manifest, NP+DI raw catches 6 of 13 actually-exploited events; NP+DI+DQ catches 9; hacker S+A catches 10. Their union catches 11. Same story, different angle of attack on the same triage decision. Detail on the periodicity page.

3e. The hidden cost of not filtering: churn crowds out better practices

The case for any filter — NP+DI, hacker S+A, or both — isn't just that it reduces rebuilds from 14 a year to 2-6. It's that the rebuild churn at "all C/H" volumes actively prevents teams from adopting the practices that would make patching cheaper.

Teams fighting a fire every two weeks don't typically use that bandwidth to fix root causes. They don't get to "let's adopt a get-latest build model so the next CVE is a non-event." They don't get to move infrastructure from reactive (fix when broken) to cadence-based (regular updates absorb most patches). The natural reflex is "we should just fix things right" — but humans in firefighting mode don't behave that way; they triage, ship, and brace for the next alert.

The noise-floor reduction from 14 forced rebuilds per year to 2-6 isn't just a workload win — it's the difference between a team that only does emergency rebuilds and a team that has time to invest in not having to. We pick this up again in §9 with the Cat 1/2/3 estate-maturity framing.

4c. Silence Windows

None of these strategies produce a slower steady drip — they all produce silence punctuated by genuine events. Spring Boot's longest NP+DI silence ran 164 days; hacker S+A stretches that to 205. Node.js sat silent under NP+DI for 311 days — over 10 months — before its first trigger. Even Django, the weakest case, had a 140-day NP+DI silence window. This burst pattern is operationally superior to a steady cadence: when the signal fires, your team responds to it instead of tuning it out.

4d. What the Filter Caught

Spring Boot (5 events): Spring Security authorization bypass (CWE-693, CWE-863) via annotation detection failure, Tomcat CLIENT_CERT authentication bypass (CWE-287), Tomcat path traversal (CWE-23) via crafted HTTP request, and Thymeleaf template injection (CWE-1336/917) -- textbook SSTI leading to RCE. The auth bypass events reflect the widened DI definition: untrusted input that changes a security decision.

Node.js (2 events): Koa host header injection (CWE-74) enabling cache poisoning, and Handlebars code injection (CWE-94) via AST type confusion -- four separate attack vectors in the same disclosure. Template engine code injection -- same vulnerability class as the Spring Thymeleaf hit.

Django (4 events): All four were SQL injection (CWE-89) in Django's ORM layer itself -- recurring variants of the same underlying issue in column alias handling and QuerySet connector kwargs. The filter correctly flagged every one.

4e. Django: The Honest Hardest Case

Django is the weakest result, and that's worth understanding. Unlike Spring and Node.js (which had long silence windows), Django produced 4 NP+DI triggers in 12 months -- roughly quarterly. All four were CWE-89 in Django's ORM. The Django security team kept finding and patching variants of the same underlying issue.

This means the filter gives you less breathing room when the framework itself has a persistent vulnerability pattern in its parsing surface. But: (a) every trigger was a genuine SQL injection -- zero false positives, and (b) 71% reduction is still operationally meaningful -- you go from near-monthly fire drills to quarterly ones. The 10 skipped rebuilds were DoS bugs, config issues, and non-NP library fixes that never reached KEV.

Why Most Criticals Don't Get Exploited

In plain English: 93% of "critical" vulnerabilities never get used in real attacks. The reasons are predictable -- and that predictability is what makes smart prioritization possible.

Six patterns explain why the other 93% of critical/high CVEs don't reach KEV — but some of these carry important caveats:

Firmware and hardware (3,739 NVD, 23 KEV, 0.62%). High CVSS because theoretical impact is complete system compromise, but exploitation requires physical access and specific device models. The 23 windowed KEV entries are mostly Qualcomm and Arm silicon bugs, BIOS/UEFI issues, and BMC vulnerabilities.

Linux kernel local privilege escalation. Many kernel CVEs score critical but require local access. These aren't initial-access vectors — they're land-and-expand tools. An attacker who already has a foothold (via a network-parser bug, a phished browser, or stolen credentials) uses kernel privesc to escalate. That makes them dangerous, but in a fundamentally different way: they determine how bad a breach gets, not whether one starts. We cover this distinction in §8.

Consumer IoT and embedded devices. High CVSS, low strategic value. Nation-state actors focus on enterprise infrastructure.

Memory corruption complexity — with a major exception. Server-side heap overflows and use-after-free bugs score 9.8 but require precise heap grooming, specific binary versions, and ASLR/DEP bypasses. The complexity genuinely filters out most opportunistic attackers. Browser UAF is the exception. Chrome and WebKit use-after-free bugs get weaponized within days — sometimes hours — by exploit brokers and state-sponsored actors. The difference is economics: a server-side UAF compromises one target at a time; a browser UAF delivered via phishing or watering-hole can reach millions. The browser exploit market is liquid enough that complexity isn't a barrier.

Open-source library usage variance. A library bug is exploitable only when the application uses the library in the specific way the bug exposes — and library APIs are large, optional, and vary wildly across apps. Jackson-databind has accumulated 67+ historical CWE-502 entries (deserialization → RCE), each technically critical because remote code execution is theoretically possible. Almost all stay unreachable because exploitation requires three things to line up: the target gadget classes loaded on the classpath, the unsafe deserialization API in the application's request handling path, and attacker-reachable input that gets routed there. Most apps have at most one. Same shape elsewhere: lodash _.template code injection only fires if the app passes attacker-controlled template strings to it (most don't); SnakeYAML constructor RCE only fires if the app calls yaml.load() on attacker bytes instead of yaml.safe_load(); Pillow image bugs depend on which API path the app routes uploads through. The exception that proves the rule: network-parser libraries (Tomcat's HTTP handling, Spring's MVC layer, log4j's logging API) are exploited at much higher rates — precisely because their usage pattern is uniform across applications. Every Tomcat install handles HTTP the same way. Most other libraries don't, and a generic exploit can't depend on a surface that varies app to app.

Non-Western ecosystem and niche products. Products with limited Western enterprise deployment accumulate NVD criticals but have minimal KEV representation.

Implication: These patterns are predictable, but context-dependent. Server-side memory corruption really does stay unexploited most of the time. Browser memory corruption doesn't — it's one of the fastest-exploited categories in the dataset. And kernel privesc bugs aren't "safe to ignore" just because they're not initial access; they're the second half of the kill chain. The right question isn't "will this get exploited?" but "where does this sit in the attack sequence, and does my architecture already contain it?"

8a. WAF defensibility on the actually-exploited events

In plain English: Some bugs a WAF can hold the line on for days while you patch. Others a WAF cannot meaningfully help with at all. That distinction tells you which patches genuinely have to ship fast.

Putting It Together: Operational Response by Estate Maturity

In plain English: The filter tells you which events to treat as emergencies. But what counts as an "emergency response" depends on what your team can actually do — a team that ships in hours has a different threshold than one that ships in months. The right operational frame sorts by estate maturity, not by trigger type.

The honest version of "what should you do about this" depends on a parameter the filter alone doesn't capture: your fastest credible rebuild SLA. A team that can absorb a security patch in hours via routine BAU rebuild has very different operational needs than a team where every rebuild is a multi-week project. The same NP+DI trigger creates very different responses across those organizations.

The reduction in rebuild churn isn't just a workload win; it's what unlocks the structural fixes.

Two operational rules apply in all three categories:

OS container: refresh on a regular cadence regardless of CVE pressure. The OS layer never generates emergency rebuild triggers under either NP+DI or hacker S+A (§7). But unpatched containers accumulate local privilege-escalation chains. Monthly refresh keeps worst-case post-compromise exposure at ~5-10 CVEs instead of 21. This is blast-radius control, not emergency response.

Everything else rides the team's normal update process. Non-NP library updates, non-DI C/H vulnerabilities, DoS bugs, info leaks. Zero of 34 skipped app-layer events and none of 48 OS-layer NETWORK events in our 12-month window appeared in KEV. They're not urgent. They get caught up in whatever cadence the team already has — cycle-time-fast for Cat 1, regular for Cat 2, reactive for Cat 3.

The shift from the old model: The old 72h/7d/30d SLA framework treated urgency as a function of CVSS severity and KEV presence. The new model treats urgency as a function of attack surface structure (where the vulnerability sits in the parsing stack) and estate maturity (what the team can credibly do about it). Filtering by structure alone produces 2-6 genuinely dangerous events per year — but whether that's "manageable cadence" or "still a crisis every two months" depends on which Cat the team is in. The two questions interact.

9a. The WAF Dividend

There's a useful byproduct of reducing emergency triggers to 2-6 per year: at that volume, targeted WAF rules become practical.

At "all critical/high" scale — thousands of CVEs per year — maintaining custom WAF signatures is operationally impossible. Nobody writes and tests custom rules at that cadence. At "all network parsers" (14 triggers/year) it's theoretically possible but the rule diversity is enormous: DoS patterns look nothing like injection patterns look nothing like authentication bypasses. At NP+DI (2-6/year), you're writing a handful of targeted rules for the bug classes WAFs are specifically designed to catch.

And this is where the "DI" half of the filter pays a second dividend. Direct injection attacks are structurally visible in the request: a ../ in a path parameter, a '; DROP TABLE in a query string, a ${7*7} in a template field, a ; cat /etc/passwd in a header value. WAF vendors have spent two decades refining signatures for exactly these patterns. When a new path-traversal CVE drops in Tomcat, a WAF rule that blocks ../ sequences in the relevant URL path is straightforward to write, test, and deploy — often buying days or weeks of protection while you prepare the actual rebuild.

Contrast this with deserialization attacks, where the payload is typically an opaque serialized object blob (Java ObjectInputStream, Python pickle, PHP unserialize). A WAF would need to actually deserialize the payload to understand its intent — which is exactly the vulnerability. WAFs can flag known gadget chain byte signatures, but novel chains pass through because the malicious semantics are invisible at the HTTP layer. This is one reason the DI CWE set deliberately focuses on injection classes where the attack structure is visible in transit, not on deserialization (CWE-502), which is genuinely harder to mitigate at the edge.

The practical upshot: When a Tier 1 trigger fires, your response isn't just "rebuild." It's "deploy a targeted WAF rule within hours for immediate mitigation, then rebuild on your timeline." The NP+DI filter gives you few enough events that maintaining those rules is realistic, and the DI bug classes are precisely the ones WAFs handle well.

A caveat: WAFs are not a substitute for patching. Determined attackers find bypass techniques — encoding tricks, edge cases, double-encoding — and WAF rules for a specific CVE are only as good as the researcher's understanding of the exploit path. The value is in raising the bar against opportunistic and automated exploitation, which is exactly what most KEV-listed activity actually is. The WAF buys time; the rebuild closes the hole.

The Reverse Proxy Myth

In plain English: "Our apps sit behind a reverse proxy" doesn't protect against application-layer exploits. Spring4Shell and Log4Shell both pass through proxies untouched.

A common pushback: "Our apps sit behind a reverse proxy, so library vulnerabilities are mitigated." This is partially true for network-layer attacks but largely false for application-layer exploits. Spring4Shell and Log4Shell both pass through mod_proxy untouched -- the proxy doesn't parse the application-layer semantics of what it forwards.

A WAF is a different story. As discussed in §9a, WAFs are genuinely effective against the DI attack classes — but they're a time-buying measure, not a permanent fix. A targeted WAF rule deployed hours after a CVE drops can block the known exploit path while you prepare your rebuild. The reverse proxy underneath, however, is just a pipe: it forwards whatever the WAF doesn't catch, and it doesn't understand what it's forwarding.

This is a companion to the three-tier patching model: the proxy constrains what the attacker can send to the parsing surface, the WAF filters known attack patterns, and the NP+DI filter tells you when to actually rebuild. Each layer has a different job.

Reverse proxy defense-in-depth: what actually works (expand for guidance)

Path constraint / URI allowlisting. The single most effective control. If your app only serves /api/v1/* and /static/*, reject everything else at the edge. Spring4Shell's exploitation relied on accessing class-loader paths -- a path allowlist blocks this even without knowing the CVE.

Request size and depth limits. Cap body size, header count, header size, query string length, and URL depth. Many deserialization payloads require oversized request structures.

Header stripping and normalization. Strip headers your app doesn't need. If your backend doesn't use X-Forwarded-For, strip it -- this would have neutered one of Log4Shell's injection vectors.

TLS termination. Without it, the proxy is just a TCP forwarder and can't apply any of the above controls.

Response header hardening. Remove server version banners, framework identifiers, and stack traces from error responses.

The practical takeaway: A proxy configured with path allowlisting + header stripping + request size limits constrains what the attacker can send to a surface small enough that many exploit payloads can't reach the vulnerable code path. The key word is "constrain," not "prevent." It's a seatbelt, not an airbag.

Caveats & Methodology Limitations

In plain English: This analysis has real blind spots. We're transparent about what the data can and can't tell us.

KEV is government-centric and network-biased. CISA KEV is curated for US federal agency compliance. It captures exploits that hit government networks well. It captures browser exploitation, phishing-delivered exploits, and commercial-sector targeting less well. Our exploitation rates for "local access" components are likely understated.

Supply-chain exploitation is invisible to KEV. A critical vulnerability in axios (100M+ weekly downloads) that gets patched within hours may never appear in KEV. These invisible library updates are the daily operational burden this analysis aims to help prioritize.

NVD denominator quality. CPE mapping is imperfect. Approximately 17,000 CVEs fell into "other" due to sparse metadata.

Library detection limits. NVD's CPE-based categorization dramatically undercounts library vulnerabilities. We use NVD as the primary denominator for cross-layer comparisons and supplement with OSV for library-specific analysis.

Time-to-exploit right-censoring. Recent years show artificially higher medians because many CVEs haven't been added to KEV yet.

Network-parser classification is a judgment call. The binary parser/non-parser split is a simplification. The CVE reference list shows every classification so you can audit the line.

Periodicity-Specific Caveats

OSV severity gaps. ~26% of OSV advisories lack parseable severity data. Some C/H events may be missed, though major ones with GHSA or NVD backing are captured.

CWE classification is imperfect. CWE-20 ("Improper Input Validation") is particularly ambiguous -- some vulns classified as "not direct injection" might have injection characteristics under specific conditions.

KEV lag. Zero KEV entries for these manifests in 12 months doesn't mean none of these vulns matter -- it means CISA hasn't observed confirmed exploitation yet. The filter is predictive, not retrospective.

Transitive depth. Real apps pull far more transitive dependencies than modeled (Spring: 200+ JARs, Node.js: hundreds of npm packages). The full tree would push all-C/H periodicity even shorter while NP+DI events would grow proportionally less.

Django's ORM SQLi pattern may not generalize. Django happened to have a persistent SQL injection issue during this 12-month window. A different window might show a different pattern.

12-month window. Twelve months of "zero misses" is strong but not conclusive. A 24-month retest with a wider manifest set would either strengthen the case or surface the first miss. The goal isn't to be exactly right; it's to be slightly conservative and never wrong.
